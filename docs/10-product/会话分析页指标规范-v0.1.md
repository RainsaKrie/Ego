# Ego `v0.1` 会话分析页指标规范

## 1. 文档定位

本文件定义 `会话分析` 页面在 `v0.1` 中应展示哪些指标、这些指标的优先级如何，以及哪些内容不应在首版做得过重。

`会话分析` 页面不是把工作台右栏放大，也不是全局 BI 看板。
它的职责是服务“单个会话的复盘”。

大白话解释：
工作台更像驾驶过程中的仪表，分析页更像停下来以后看整段行程报告。

## 2. 页面目标

`v0.1` 的 `会话分析` 页面需要同时完成四件事：

1. 让用户看清这场会话整体跑成什么样
2. 让用户能发现 Token、延迟、成本的变化趋势
3. 让用户能观察记忆策略和参数变化带来的影响
4. 不把页面做成一个复杂难懂的数据后台

## 3. 核心原则

### 3.1 回答问题，而不是堆数字

每个指标都必须对应一个真实问题。

例如：

- 请求数：这场会话到底做了多少轮实验
- 累计 Token：这场会话到底吃了多少上下文资源
- 累计成本：这场会话大概花了多少钱

如果一个指标不能清楚回答某个问题，它就不应在首版占主要位置。

### 3.2 会话级优先，不做全局 BI

`v0.1` 的分析页只先服务“单个会话的复盘”。

当前不做：

- 全局跨会话对比仪表盘
- 多模型横向实验总览中心
- 长周期历史趋势仓库

### 3.3 趋势要轻，解释要强

分析页可以有趋势信息，但首版不应过度依赖复杂图表。

因此首版优先：

- 清晰指标
- 简单趋势
- 明确解释

### 3.4 单轮与整场要分层

工作台右栏已经负责单轮请求观测。

分析页负责：

- 汇总
- 比较
- 复盘

不能把分析页做成“最近一次请求详情页”。

## 4. 指标分层

### 4.1 第一层：会话总览指标

这是分析页最上层，也是优先级最高的一组指标。

建议展示：

- 请求总数
- 累计 Token
- 累计成本估算
- 平均延迟

当前正式结论：

- `请求总数`
- `累计 Token`
- `累计成本估算`

是首版最核心的总览指标。

`平均延迟` 可以进入首屏总览，但优先级低于前三项，更适合作为整体体感指标，而不是核心主指标。

补充约束：

- `累计成本估算` 必须明确标注为“估算”
- 它不能脱离模型价格表与 Token 构成，被当成一个孤立数字展示

### 4.2 第二层：资源构成指标

这层指标不是看总量，而是看“资源是怎么被消耗的”。

建议展示：

- Prompt Token 总量
- Completion Token 总量
- 输入/输出 Token 占比

这组指标回答的问题是：

- 是历史上下文把 Token 撑高了
- 还是模型回复太长把 Token 撑高了

补充约束：

- 如果首屏展示 `累计成本估算`，页面中必须存在与其对应的解释支撑
- 至少应能看见当前模型、价格表依据，或输入/输出 Token 构成中的一种以上信息

### 4.3 第三层：趋势型指标

这层指标进入“变化过程”的观察。

建议关注：

- 每轮 Token 变化趋势
- 每轮延迟变化趋势
- 每轮成本估算变化趋势

当前正式结论：

- `Token 趋势` 是首版必须具备的主趋势
- `延迟趋势` 建议进入首版
- `成本趋势` 可以进入首版，但不应与前两者同权重强调

大白话解释：

- `Token 趋势` 是资源主线
- `延迟趋势` 是体验主线
- `成本趋势` 是经济解释主线

三者相关，但首版不应被当成完全同权。

首版建议：

- 不做复杂交互图表
- 可用简单折线或条形趋势
- 不做复杂分析系统

### 4.4 第四层：实验条件变化记录

这是分析页最能体现实验复盘价值的一层。

建议关注：

- 模型是否发生切换
- 记忆策略是否发生切换
- 关键参数是否发生明显变化

当前正式结论：

- 首版要做
- 但应保持低存在感
- 只在发生关键实验条件变化时发挥作用

补充约束：

- 在真实使用中，单次会话内部通常不会频繁切换模型或大幅改参数
- 因此这一层不应被设计成高密度事件时间轴
- 如果整个会话未发生关键实验条件变化，这一块应保持很弱，甚至只显示“未发生关键实验条件变化”
- 它的目标是解释趋势拐点，而不是制造事件噪音

## 5. 首版页面结构建议

建议 `会话分析` 页面从上到下分为 4 段：

1. 会话总览区
2. 资源构成区
3. 趋势区
4. 实验条件变化记录区

### 5.1 会话总览区

建议放：

- 请求总数
- 累计 Token
- 累计成本估算
- 平均延迟

### 5.2 资源构成区

建议放：

- Prompt Token 总量
- Completion Token 总量
- 输入/输出占比

### 5.3 趋势区

建议放：

- Token 趋势
- 延迟趋势
- 成本趋势（弱化）

### 5.4 实验条件变化记录区

建议放：

- 当前会话中是否发生过模型切换
- 是否发生过记忆策略切换
- 是否发生过关键参数变化

这块不一定要很花哨，但至少要让用户知道：

- 这场会话是否在完全固定的条件下跑完

## 6. 首版不建议做的内容

以下内容不建议在 `v0.1` 首版做太重：

- 跨会话对比大盘
- 自动结论生成
- 复杂筛选器
- 多图联动分析
- 细粒度到每个 token 的全局可视化

原因：

- 会显著增加复杂度
- 会偏离当前的学习目标
- 会让页面更像数据后台，而不是实验复盘页

## 7. 当前定案的首版最小集

如果需要进一步压缩首版范围，至少应保留：

- 请求总数
- 累计 Token
- 累计成本估算
- Prompt / Completion Token 构成
- Token 趋势

这组内容已经足以支撑 `v0.1` 的核心学习目标。

## 8. 当前定案结论

`v0.1` 当前正式结论如下：

- 分析页的主目标是会话级复盘
- 首屏先看总量，不先看复杂图表
- Token 相关指标是一等公民
- `累计成本估算` 进入首屏总览，但必须明确是估算，且需要解释支撑
- `平均延迟` 可进入首屏总览，但优先级低于前三项
- 趋势区首版以 `Token 趋势` 为主，`延迟趋势` 为辅，`成本趋势` 弱化
- 实验条件变化记录区首版要做，但保持低存在感，只在关键变化发生时发挥作用
