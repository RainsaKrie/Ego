# Ego `v0.1` 信息架构

## 1. 文档定位

本文件用于定义 `Ego v0.1` 的页面层级、信息优先级和核心交互路径。

这里讨论的不是视觉风格，而是产品结构。

大白话解释：
谁出现在一级页面、谁常驻可见、谁被放进折叠区，决定了这个产品真正重视什么。

## 2. 信息架构目标

`v0.1` 的信息架构需要同时满足以下目标：

1. 让对话保持主入口地位
2. 让参数调优和运行观测成为一等公民
3. 不把产品做成一个普通聊天壳
4. 为未来知识层、工具层和 Agent 层保留扩展空间

## 3. 核心原则

### 3.1 对话是主流程，不是唯一主角

用户的主要动作仍然是发起对话、查看回复、继续追问。

但 `v0.1` 不是只关心“回答内容”，还关心：

- 花了多少 Token
- 上下文如何变化
- 当前记忆策略是否带来副作用
- 参数变动是否真的改变了输出

因此，对话区不能独占全部产品重心。

### 3.2 观测能力必须低门槛可达

核心观测信息不能藏在很深的设置页里。

至少以下内容应在主工作台中可快速看到：

- 当前模型
- 当前 `temperature`
- 当前 `top_p`
- 当前记忆策略
- 最近一次请求的 Token / 延迟 / 成本

### 3.3 复杂信息可以折叠，但不能消失

`v0.1` 不应一上来把所有数据全部摊开，否则会让界面过重。

因此需要区分：

- 常驻可见信息
- 一键展开信息
- 专门分析页中的深度信息

### 3.4 页面结构必须服务实验闭环

产品结构要服务这样一条动作链：

1. 提问
2. 调参
3. 发送
4. 观察结果
5. 理解差异
6. 继续实验

### 3.5 参数控制必须分层暴露

`v0.1` 的参数既不能全部塞进主界面，也不能全部藏起来。

应遵循：

- 高频实验参数放在主工作台常驻区
- 高级参数放入折叠区
- 提供方专属参数暂不进入通用主界面

## 4. 一级页面

`v0.1` 只保留 3 个一级页面：

1. `对话工作台`
2. `会话分析`
3. `全局设置`

这个数量是刻意控制的。

原因：

- 页面太多会打散主线
- 页面太少会把分析能力挤回抽屉

## 5. 页面职责

### 5.1 对话工作台

这是 `v0.1` 的主入口和主战场。

目标：

- 承载日常对话
- 承载参数调整
- 承载单轮请求级观测

建议结构：

- 左侧：会话列表
- 中间：聊天主区
- 右侧：会话参数与请求观测侧栏

左侧会话列表负责：

- 新建会话
- 切换会话
- 重命名会话
- 快速查看会话标题与最近更新时间

中间聊天主区负责：

- 展示消息流
- 输入 Prompt
- 发送请求
- 查看回复

右侧侧栏负责：

- 当前模型配置
- 当前记忆策略
- 高频参数调节
- 最近一次请求的 usage 摘要
- 本会话的轻量统计

已确认约束：

- 右侧侧栏采用常驻方案
- 最近一次请求观测必须在此页面稳定可见

大白话解释：
这个页面不是“聊天区旁边顺手放点设置”，而是“聊天、调参、看结果”并列发生。

### 5.2 会话分析

这是 `v0.1` 的实验复盘区，并且应作为一级页面独立存在。

目标：

- 不再只看单轮请求
- 而是看一个会话在多轮交互中的累计变化

首版建议展示：

- 请求次数
- 总 Token 消耗
- 输入输出 Token 分布
- 平均延迟
- 累计成本估算
- 记忆策略对上下文增长的影响

后续可以扩展：

- 同会话内不同参数阶段的比较
- 同类 Prompt 的对比观察

但这些不要求在 `v0.1` 首版一次性做满。

### 5.3 全局设置

目标：

- 承载全局默认规则
- 不打断主实验流程

建议放入：

- API Base URL
- API Key
- 默认模型
- 默认参数
- 默认记忆策略
- 价格表配置

原则：
设置页负责定义默认环境，而不是承担日常实验动作。

## 6. 对话工作台的信息优先级

### 6.1 必须常驻可见

以下信息建议在主工作台常驻可见：

- 当前会话标题
- 当前模型
- 当前 `temperature`
- 当前 `top_p`
- 当前 `max_output_tokens`
- 当前记忆策略
- 最近一次请求的 Token 总量
- 最近一次请求的延迟
- 最近一次请求的成本估算

### 6.2 可以折叠展开

以下信息可以放入折叠区或二级面板：

- Prompt Token / Completion Token 拆分
- 上下文实际包含了多少轮历史
- usage 数据来源是精确返回还是本地估算
- 本轮请求的完整参数快照
- `seed`
- `top_logprobs` 结果展示
- `truncation`

### 6.3 适合放到分析页

以下内容更适合放到分析页：

- 一段时间内的 Token 增长趋势
- 会话累计成本变化
- 不同记忆策略的统计对比
- 更长时间尺度的实验记录

## 7. 会话级覆盖状态

当当前会话的参数偏离全局默认值时，应给出轻提示。

建议方式：

- 在会话标题旁或参数区顶部显示轻量标签
- 文案可以类似 `已覆盖默认配置`
- 点击后可查看偏离项明细

提示内容至少应帮助回答：

- 哪些参数偏离了默认值
- 默认值是什么
- 当前值是什么

原则：
这是实验环境提示，不是警报。

## 8. 最短核心路径

`v0.1` 至少应支持一条顺畅的最短路径：

1. 创建会话
2. 配置模型和参数
3. 发送 Prompt
4. 得到回复
5. 立刻看到该轮请求的关键观测数据
6. 继续调整参数并再次发送
7. 在分析页复盘整个会话

如果这条路径不顺，说明信息架构存在问题。

## 9. 边界控制

本期不纳入以下一级页面：

- 知识库页面
- Agent 页面
- 工作流页面
- 工具市场页面
- 系统级自动化控制台

原因不是它们不重要，而是它们会在当前阶段冲散主线。

## 10. 当前定案结论

`v0.1` 当前正式结论如下：

- 一级页面保持 3 个
- `对话工作台` 是主入口
- `会话分析` 作为一级页面独立存在
- `全局设置` 只承载默认规则
- 观测能力必须在主工作台中稳定可见
- 右侧侧栏先采用常驻方案
- 最近一次请求观测位于主界面一等位置
- 高频参数直接展示
- 高级参数进入折叠区
