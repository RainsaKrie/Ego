# Ego `v0.1` 产品需求文档

## 1. 产品定义

### 1.1 愿景

Ego 的长期目标是成为一个“个人 AI 操作系统”。

大白话说，它不只是一个聊天框，而是一个能长期承载 AI 对话、记忆、未来智能能力和个人工作流的统一工作台。

### 1.2 `v0.1` 产品形态

`v0.1` 不是完整的 AI OS。

`v0.1` 的定位是一个 AI 对话实验台，特点如下：

- 产品思路坚持本地优先
- 默认采用 `BYOK`
- 重点服务于“理解模型行为”和“做实验”
- 不隐藏 Token、上下文和参数影响，而是把它们直接暴露出来

### 1.3 核心目标

帮助产品负责人真正理解下面几件事：

- `temperature` 如何影响输出稳定性和创造性
- 上下文长度如何影响回答质量、成本和延迟
- 短期记忆策略如何影响对话连续性
- Token 消耗如何被监测、分析和优化

## 2. 目标用户

当前阶段的核心用户就是产品构建者本人：

- 对技术和模型机制有强烈兴趣
- 愿意主动实验 Prompt 和模型参数
- 希望有一个日常可用的 AI 工作台，而不是散落在浏览器里的多个网页标签
- 希望把理论认知真正变成产品直觉

未来也可能面向更高级的个人创作者或开发者，但这不是 `v0.1` 的首要目标。

## 3. 核心使用场景

### 场景 A：结构化对话实验

用户开启一个会话，选择模型，调整 `temperature`，发送 Prompt，并观察输出质量和成本的变化。

### 场景 B：查看运行成本

用户希望知道：

- 每一轮对话消耗了多少 Token
- Prompt 和 Completion 各自占了多少
- 不同模型或不同记忆策略的成本差异有多大

### 场景 C：比较不同记忆策略

用户在相似对话下切换不同记忆策略，观察：

- 回答是否还能保持连续性
- Token 是否持续膨胀
- 延迟是否明显增加
- 摘要记忆是否真的能控制成本

## 4. `v0.1` 范围

### 4.1 本期要做

- 桌面优先的个人应用
- 多个聊天会话
- OpenAI 兼容 API 配置
- 会话级模型配置
- 请求与响应历史保存
- 当提供方返回 usage 数据时进行 Token 追踪
- 当提供方未返回 usage 数据时进行本地估算兜底
- 延迟追踪
- 成本估算
- 基础记忆策略切换
- 轻量级的会话与请求可观测性视图

### 4.2 本期不做

- 多 Agent 系统
- 自主规划循环
- 工作流搭建器
- 浏览器自动化
- 本地知识库注入
- 插件市场
- 团队协作
- 云同步

## 5. 功能需求

### 5.1 对话能力

- 用户可以创建、重命名和切换聊天会话。
- 用户可以发送 Prompt 并获得模型回复。
- 系统必须在本地保留消息历史。

### 5.2 模型配置

- 用户可以配置 API Base URL、API Key 和模型名称。
- 用户可以为每个会话调节最小必要参数集。
- 初始参数集应故意保持精简，避免 UI 一开始就过重。

建议首批参数只包含：

- `model`
- `temperature`
- `max_completion_tokens` 或等价的输出长度限制

### 5.3 记忆实验

系统至少应支持以下几种记忆策略：

- `none`：只发送当前这次用户输入
- `recent-window`：只携带最近 N 轮对话
- `summary-plus-recent`：携带滚动摘要和最近 N 轮对话

大白话解释：
所谓记忆策略，本质上就是“下一次请求时，到底带上哪些旧对话内容”这一套规则。

### 5.4 可观测性

系统应尽量为每次请求记录：

- 请求时间
- 使用的模型
- 生效的记忆策略
- Prompt 大小
- Completion 大小
- 总 Token 消耗
- 延迟
- 预估成本
- usage 数据来自提供方还是本地估算

### 5.5 比较价值

产品应能帮助回答类似这样的问题：

- 把 `temperature` 调低之后，输出是否更稳定
- 摘要记忆是否降低了 Token 膨胀
- 哪个模型的“单位成本产出质量”更高

## 6. 非功能需求

- 默认本地优先
- UI 行为可预测、易理解
- 不允许后台悄悄改写用户内容
- 当提供方 usage 数据缺失时要优雅降级
- 对估算值的不确定性要有清晰标识

## 7. 风险

### 7.1 提供方返回格式不一致

不同 API 对 usage 字段的返回方式并不一致，尤其是在流式场景中更明显。

影响：
系统必须明确区分“提供方精确返回的数据”和“本地估算的数据”。

### 7.2 范围膨胀

产品方向很大，但 `v0.1` 必须刻意保持克制。

影响：
任何 Agent、工作流或大量工具接入相关内容，除非再次明确批准，否则都不进入当前里程碑。

### 7.3 记忆概念混淆

用户很容易把对话历史、摘要记忆、知识库召回混成一件事。

影响：
`v0.1` 只解决“短期对话记忆”，不解决“长期知识记忆”。

## 8. 成功标准

如果满足以下条件，说明 `v0.1` 达到目标：

- 产品负责人愿意把它当成真实聊天工作台使用
- 每次请求的 Token、成本和延迟都能被看到
- 不改代码也能实验不同记忆策略
- 能从中获得关于 Prompt 与运行参数的真实直觉

## 9. 延后议题

以下内容有意延后：

- 长期知识库整合
- Agent 运行时
- 工作流引擎
- 工具调用框架
- 超出基本兼容层之外的多提供方抽象
