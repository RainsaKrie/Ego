# Ego `M2` 当前验收与缺口清单

## 1. 文档定位

这份文档不是新的产品规范，也不是新的架构设计。

它只解决一个现实问题：

`当前 M2 已经推进到哪里了，还差哪些点，什么时候可以认为这一阶段可以收口。`

大白话说：

它是一份阶段性验收清单，不是长期原则文档。

## 2. 当前对 `M2` 的理解

根据已经生效的里程碑与实现边界，`M2` 的目标不是分析页，也不是更大的产品扩展，而是：

- 让对话从 `M1` 的“主链可跑通”
- 升级到更像真实 AI 工作台的“生成中可感知、可中止、失败可补救”

因此，`M2` 当前主要围绕 3 条主线：

1. 真实流式输出
2. 生成过程控制
3. 失败后的轻量恢复

## 3. 当前已完成项

截至当前仓库状态，`M2` 已完成这些核心能力：

### 3.1 真实流式输出

- 后端已接入真实流式 Provider 请求
- 前端已显示临时助手输出
- 只有完成后才落正式助手消息
- 流式失败时不把半截内容转成正式消息

### 3.2 手动停止生成

- 流式过程中，发送按钮会切换为中止按钮
- 用户可主动结束当前生成
- 中止后的请求会进入 `cancelled`
- 中止后的半截输出不会落成正式助手消息

### 3.3 失败后轻量重试

- 最近失败的一轮请求可直接轻量重试
- 重试不会重复插入用户消息
- 重试会生成新的 `RequestRecord`
- 旧失败记录会继续保留

### 3.4 流式过程中的基础观测补强

- 右栏最近一次请求在流式进行中可实时显示当前耗时
- 流式未完成前，`Token / 成本 / 来源` 不再假装已有最终值
- 完成后再切换为正式 usage 信息

### 3.5 失败与重试的交互收口

- 消息区保留轻量失败提示
- 输入区保留轻量重试入口
- 失败提示已减重，不再用重复说明打断主流程

## 4. 当前已满足的 `M2` 验收标准

如果只看主链能力，当前已经满足：

1. 用户能看到真实流式文本逐步出现
2. 用户能中途停止当前生成
3. 停止后不会留下正式半截回复
4. 流式失败后能快速重试
5. 失败记录与重试记录不会互相覆盖
6. 最近一次请求在流式中具备最小可观测性

也就是说：

`M2` 的核心运行时闭环已经基本形成。`

## 5. 当前仍存在的缺口

虽然主链已基本成立，但当前仍有几类缺口没有完全收口。

### 5.1 观测仍然偏“工作台级”，还不是“分析级”

当前已经有：

- 最近一次请求
- 流式过程中的实时耗时

但还没有：

- 更深入的失败场景对比视图
- 更系统的会话级实验复盘支撑

这类能力更接近后续分析页与 `M3`，不一定属于当前 `M2` 阻塞项。

### 5.2 失败恢复仍是轻量版

当前重试已经成立，但仍然刻意保持克制：

- 只支持最近失败的一轮
- 不支持任意历史请求重试
- 不支持断点续写
- 不支持未完成消息保留

这不是缺陷，而是当前阶段主动控制复杂度的结果。

### 5.3 交互仍有进一步减重空间

当前主链已可用，但某些提示、占位、说明和默认文案仍可能继续减重。

这类问题更像 UI 收口事项，而不是 `M2` 主链阻塞项。

## 6. 哪些不算 `M2` 未完成

为避免范围再次发散，下面这些不应被误判为“`M2` 还没做完”：

- 分析页做深
- 会话级复杂统计视图
- 多提供方高级适配
- 边流边精确 token 统计
- 默认环境预设系统
- 更复杂的知识库或 Agent 能力

这些不是 `M2` 的收口条件。

## 7. 当前建议的 `M2` 收口标准

我建议把 `M2` 是否可以收口，收束成下面 3 个判断：

### 7.1 主链稳定

至少要确认这些真实使用行为稳定成立：

- 正常流式完成
- 中途手动中止
- 流式失败
- 失败后轻量重试

### 7.2 观测可信

至少要确认：

- 流式中实时耗时正确增长
- 完成后最终 usage 正确替换
- 失败或取消时不伪造最终 usage

### 7.3 交互不再明显打架

至少要确认：

- 消息区、输入区、右栏之间不再互相挤压
- 失败提示、重试动作、发送/中止动作的关系已经自然
- 不再存在明显误导用户的状态表达

## 8. 当前推荐结论

基于目前仓库状态，我的判断是：

- `M2` 核心主链已经基本完成
- 当前更像“进入收口阶段”
- 而不是“还差一条大的主能力没做”

也就是说，接下来最合理的方向不是继续横向扩功能，而是：

1. 做一轮真实使用验收
2. 记录仍然存在的交互或状态问题
3. 再决定是否宣告 `M2` 收口，进入 `M3`

## 9. 当前文档结论

这份清单当前的作用只有一个：

把 `M2` 从“感觉差不多了”变成“有标准地判断差不多了”。

如果后续再出现新的 `M2` 能力扩张需求，应先判断：

它到底是在补 `M2` 缺口，
还是已经开始把范围推向 `M3` 或后续版本。
