# Ego `M1` 开发任务分解

## 1. 文档定位

本文件把 `v0.1` 的 `M1` 里程碑拆解为可执行开发任务、依赖顺序和验收口径。

它解决的核心问题是：

- `M1` 具体先做哪些任务
- 哪些任务互相依赖
- 什么结果算 `M1 Done`

大白话解释：
这份文档是把“先把发动机点火”拆成“先装哪根线、先拧哪颗螺丝”。

## 2. M1 总目标

`M1` 的目标不是做出完整首版产品，而是让以下主链第一次完整闭环：

- 能创建和切换会话
- 能配置并安全保存凭据
- 能发送一条消息
- 能拿到一条回复
- 能把消息、请求、基础 usage 正确落盘

如果这些没有全部站住，就不算真正完成 `M1`。

## 3. M1 范围边界

### 3.1 M1 要做

- 应用基础壳层可运行
- 会话列表主链
- 消息发送主链
- provider 请求主链
- `RequestRecord` 主链
- 本地持久化主链
- `API Key` 系统级安全存储
- 最近一次请求的基础观测展示

### 3.2 M1 不做

- 会话分析页
- 流式输出
- 手动停止生成
- 失败重试完整体验
- 复杂成本解释
- 高级参数折叠区的完整体验

这些不是不做，而是明确不在 `M1` 抢优先级。

## 4. 任务分组

`M1` 拆成 6 组任务。

### 4.1 G1：项目骨架与基础运行时

目标：

- 应用基础壳层能启动
- 页面导航主框架能跑
- 基础目录结构和状态边界先立住

建议任务：

1. 建立应用壳层与基础路由骨架
2. 建立核心模块目录边界
3. 建立基础状态容器与最小数据流骨架

验收口径：

- 应用能启动
- 至少能进入 `对话工作台`
- 不要求此时已经能真实对话

### 4.2 G2：凭据与全局配置主链

目标：

- 用户可以配置 API 基础信息
- `API Key` 能通过系统级安全存储管理

建议任务：

1. 实现全局设置基础表单
2. 接入系统级安全存储能力
3. 建立普通配置与敏感凭据的分层存储
4. 完成默认模型与基础参数的本地配置保存

验收口径：

- 用户可以录入 API Base URL、模型名等配置
- `API Key` 不和普通配置混存
- 关闭应用后重新打开，配置边界符合预期

### 4.3 G3：会话与消息主链

目标：

- 用户可以创建、切换、查看会话
- 用户消息能发送并落地

建议任务：

1. 实现会话创建
2. 实现会话列表与切换
3. 实现聊天消息列表基础展示
4. 实现输入区基础发送动作
5. 实现用户消息发送后立即写入

验收口径：

- 能创建多个会话
- 会话切换不混乱
- 用户消息发送后立即出现在消息流中

### 4.4 G4：请求生命周期主链

目标：

- 一次请求从发送到落地能形成最小闭环

建议任务：

1. 构建有效请求上下文
2. 在发请求前创建 `RequestRecord(pending)`
3. 调用 provider adapter 发起真实请求
4. 成功后写入助手消息
5. 更新 `RequestRecord(completed/failed)`

验收口径：

- 一次真实请求可以完整走通
- 成功与失败都能留下正确记录
- 用户消息、助手消息、请求记录之间能对齐

### 4.5 G5：本地持久化与恢复主链

目标：

- 会话、消息、请求、配置都能稳定保存

建议任务：

1. 建立本地持久化适配层
2. 持久化 `Conversation`
3. 持久化 `Message`
4. 持久化 `RequestRecord`
5. 持久化普通配置与会话覆盖值
6. 验证重新启动后的恢复行为

验收口径：

- 关闭应用再打开，核心数据仍在
- 不会把临时流式态误存成正式历史
- `pending` 请求记录能被正确保留

### 4.6 G6：基础可观测性落地

目标：

- 用户至少能看到最近一次请求的基础观测信息

建议任务：

1. 记录基础 usage 信息
2. 区分 provider-reported / locally-estimated / unknown
3. 在右栏展示最近一次请求的基础观测值

验收口径：

- 至少能看到最近一次请求的 token / latency / cost 基础信息
- usage 来源有明确区分
- 失败场景下可观测性不会完全失踪

## 5. 推荐依赖顺序

为了避免返工，`M1` 采用如下依赖顺序：

1. `G1` 项目骨架与基础运行时
2. `G2` 凭据与全局配置主链
3. `G3` 会话与消息主链
4. `G4` 请求生命周期主链
5. `G5` 本地持久化与恢复主链
6. `G6` 基础可观测性落地

大白话解释：

- 先把房子框起来
- 再装钥匙和总闸
- 再让人能进房间说话
- 再让这段说话真的能发出去
- 再确保这些事实不会关门就丢
- 最后补上基础仪表读数

## 6. 实施边界定案

当前正式结论如下：

- `M1` 允许先用非流式请求打通主链
- 非流式不是退而求其次，而是优先验证消息、请求、持久化与凭据边界的最小正确闭环
- `G2` 与 `G3` 可并行推进，但 `G2` 先于真实请求打通
- `G5` 从一开始就接真实存储适配层，不走“先纯内存后整体切换”的路线
- `G6` 在 `M1` 只要求“最近一次请求”观测，不做历史观测面板

## 7. M1 验收标准

`M1 Done` 定义为以下最小闭环全部成立：

1. 用户能通过安全存储配置并使用 `API Key`
2. 用户能创建会话并发送消息
3. 系统能拿到模型回复并显示
4. 用户消息、助手消息、请求记录都能落盘
5. 关闭应用后重新打开，核心数据仍可恢复
6. 最近一次请求的基础观测信息可见
7. 失败请求不会像没发生过一样消失

## 8. 当前定案结论

`M1` 当前正式结论如下：

- 先只打通非流式主链
- 先让消息、请求、存储、凭据边界全站住
- 不在 `M1` 抢流式体验和分析能力优先级
- `M1` 的目标不是“顺滑”，而是“可信地跑通一次完整闭环”
