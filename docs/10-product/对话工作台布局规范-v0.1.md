# Ego `v0.1` 对话工作台布局规范

## 1. 文档定位

本文件用于定义 `对话工作台` 这一一级页面的布局结构、区域职责和信息展示优先级。

这里讨论的不是视觉皮肤，而是主工作页面的权力结构。

大白话解释：
同样都是三栏布局，如果右侧观测区太弱，产品会退化成普通聊天壳；如果右侧太强，中间聊天区又会被挤压得不好用。

## 2. 布局目标

这份规范需要同时满足以下目标：

1. 保持聊天输入和阅读的顺畅性
2. 让高频参数调节足够顺手
3. 让最近一次请求观测稳定可见
4. 不让分析性信息淹没聊天主流程
5. 为桌面端后续扩展保留空间

## 3. 总体布局

`对话工作台` 采用三栏布局：

- 左栏：会话导航区
- 中栏：聊天主区
- 右栏：参数与观测区

推荐权重：

- 左栏：窄栏
- 中栏：主视觉中心
- 右栏：中等宽度常驻栏

大白话解释：
左边负责“去哪一场对话”，中间负责“正在聊什么”，右边负责“这轮是怎么跑出来的”。

## 4. 左栏：会话导航区

### 4.1 职责

左栏负责：

- 新建会话
- 切换会话
- 快速查看会话列表
- 展示会话轻量状态

### 4.2 展示内容

建议展示：

- 会话标题
- 最近更新时间
- 是否存在自定义参数覆盖的超轻量标记

首版不建议在左栏展示过多统计项，例如：

- 累计 Token
- 成本摘要
- 复杂标签体系

### 4.3 原则

- 左栏宽度保持克制
- 以“快速切换”为优先，不承担复杂控制
- 覆盖标记只负责快速扫描，不解释具体偏离项

## 5. 中栏：聊天主区

### 5.1 职责

中栏是整个页面的主视觉中心。

它负责：

- 展示消息流
- 展示模型回复
- 输入 Prompt
- 发送请求

### 5.2 为什么中栏必须最大

即使 `Ego v0.1` 是实验台，对话依然是主流程。

如果中栏被压得太窄，会直接损害：

- 长回复阅读体验
- 多轮上下文追踪体验
- Prompt 编写体验

所以右栏虽然常驻，但不能吃掉中栏的主体地位。

### 5.3 中栏分层

中栏拆为三段：

1. 顶部环境区
2. 中部消息流区
3. 底部输入区

#### 顶部环境区

建议包含：

- 一条独立的顶部信息条
- 该信息条在界面中直接命名为 `环境`
- 当前模型
- 当前记忆策略

约束：

- `环境` 信息条独立于消息流
- 不承载覆盖默认值提示主位
- 不承载参数调节
- 不承载运行统计

#### 中部消息流区

建议负责：

- 顺序展示用户与助手消息
- 保留足够留白
- 不把细碎观测数据直接塞进每条消息主体

原因：
如果把过多运行数据直接塞进消息气泡，会明显破坏阅读体验。

#### 底部输入区

建议包含：

- Prompt 输入框
- 发送按钮
- 发送前的轻量参数提醒
- 输入区下沿的会话状态读数

轻量参数提醒的作用：

- 让用户在发送前知道当前模型
- 让用户在发送前知道当前记忆策略

输入区下沿的会话状态读数规则：

- 采用弱化状态条形式
- 首版只显示 `请求数` 和 `累计 Token`
- 不放复杂趋势
- 不放成本估算

原则：
它更像当前会话的轻量生命体征，而不是一个缩小版分析面板。

## 6. 右栏：参数与观测区

### 6.1 职责

右栏负责两件事：

1. 高频参数调节
2. 最近一次请求观测

这是 `v0.1` 与普通聊天壳最关键的区别区域。

### 6.2 为什么右栏必须常驻

原因：

- 需要频繁实验参数
- 需要立刻看到结果反馈
- 如果隐藏得太深，产品就会退化成“聊天后顺便看看数据”

### 6.3 分区结构

右栏从上到下分为 3 个区块：

1. 当前会话状态与参数区
2. 高级参数折叠区
3. 最近一次请求观测区

#### 区块一：当前会话状态与参数区

建议放：

- 当前会话是否已覆盖默认配置的主提示位
- `model`
- `temperature`
- `top_p`
- `max_output_tokens`
- `memory_policy`

要求：

- 偏离默认值时，应在这里给出主提示
- 这些项应一眼可见
- 不需要多次点击才能找到

说明：

- 左栏只负责轻量扫描标记
- 右栏顶部才是解释“当前会话不是默认实验条件”的主提示位

#### 区块二：高级参数折叠区

建议放：

- `seed`
- `truncation`

原则：

- 默认折叠
- 不干扰主流程
- 需要时能快速找到

#### 区块三：最近一次请求观测区

这是右栏中优先级最高的观测区。

建议常驻展示：

- 总 Token
- 延迟
- 成本估算

建议可展开展示：

- Prompt Token / Completion Token 拆分
- usage 数据来源
- 上下文携带轮数
- `top_logprobs` 的默认 `top 3` 结果

## 7. 不建议的做法

### 7.1 不建议把右栏改成默认隐藏抽屉

因为这样会削弱 `v0.1` 的实验属性。

### 7.2 不建议把观测数据直接塞进每条消息气泡

因为这样会破坏聊天阅读体验。

### 7.3 不建议左栏承担参数调节

左栏应专注于导航，不应同时承担复杂控制。

## 8. 窄窗口适配

`Ego` 是桌面优先产品，因此正常桌面宽度下应维持三栏常驻布局。

但在窄窗口场景下，允许右栏折叠。

补充原则：

- 折叠是窄窗口适配策略
- 正常桌面宽度下，右栏仍以常驻为默认布局

## 9. 当前定案结论

`v0.1` 当前正式结论如下：

- `对话工作台` 使用三栏布局
- 左栏负责会话导航
- 中栏负责聊天主流程，并保持最大视觉中心
- 右栏负责参数调节与单轮请求观测，并保持常驻
- 左栏可显示超轻量覆盖标记，用于列表扫描
- 右栏顶部承担“已覆盖默认配置”的主提示位
- 中栏顶部不放覆盖提示，但保留独立 `环境` 信息条
- 输入区下沿放轻量会话状态读数，仅显示 `请求数` 与 `累计 Token`
- 最近一次请求观测位于右栏高优先级区域
- 会话级累计趋势不塞进主工作台，而交给 `会话分析` 页面
- 窄窗口场景下允许右栏折叠，正常桌面宽度下仍默认常驻
