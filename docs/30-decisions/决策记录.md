# Ego 决策记录

## D-000：先补齐决策协议，再推进正式选型

- 状态：已接受
- 决策：在继续深入技术选型和产品结构前，必须先建立一份明确的“架构师协议”文档，约束理解门槛、评审流程和开发准入条件。
- 原因：如果缺少这一层约束，后续很容易滑向“方案先做了、理解后补”的错误节奏，这与项目初始协作纪律冲突。

## D-001：长期愿景与 `v0.1` 形态分离

- 状态：已接受
- 决策：Ego 的长期愿景是个人 AI 操作系统，但 `v0.1` 的实际产品形态是 AI 对话实验台。
- 原因：这样既保留产品上限，又不会逼迫第一阶段一次性解决所有未来能力。

## D-002：`v0.1` 聚焦范围

- 状态：已接受
- 决策：`v0.1` 只聚焦 AI 对话、Token 可观测性、运行参数调优和记忆实验。
- 原因：这是当前最小但又最有学习价值的一组能力，足以帮助产品负责人真正理解大模型运行时取舍。

## D-003：暂缓 Agent

- 状态：已接受
- 决策：Agent 能力不进入 `v0.1`。
- 原因：Agent 会引入执行生命周期、权限边界、重试机制和失败恢复等复杂度，会干扰当前里程碑的学习目标。

## D-004：暂缓知识库注入

- 状态：已接受
- 决策：长期知识检索与知识库注入延后到 `v0.1` 之后。
- 原因：当前阶段聚焦的是对话记忆与运行可观测性，不是外部知识记忆增强。

## D-005：提供方边界

- 状态：已接受
- 决策：`v0.1` 优先支持 OpenAI 兼容 API。
- 原因：这能建立一个最常见、最实用的统一基线，也能控制提供方适配层的复杂度。

## D-006：可观测性属于产品核心价值

- 状态：已接受
- 决策：Token、延迟、成本和记忆策略影响，都是一等公民级产品能力。
- 原因：Ego 的目标不是只消费模型输出，而是帮助构建者理解模型行为。

## D-007：本地优先方向

- 状态：已接受
- 决策：Ego 从一开始就遵循本地优先方向。
- 原因：本地掌控更符合隐私、实验、以及未来桌面原生扩展的路线。

## D-008：`v0.1` 信息架构定案

- 状态：已接受
- 决策：`v0.1` 保留 3 个一级页面，分别为 `对话工作台`、`会话分析`、`全局设置`；其中右侧参数与请求观测侧栏常驻，最近一次请求观测位于主界面一等位置。
- 原因：Ego `v0.1` 的目标不是普通聊天，而是以对话为主流程、以观测与调优为并列主视图的 AI 对话实验台。

## D-009：参数分层与配置规则定案

- 状态：已接受
- 决策：`v0.1` 采用分层参数结构；`model / temperature / top_p / max_output_tokens / memory_policy` 为高频参数，`seed / truncation` 进入高级折叠区，`top_logprobs` 显示但不提供调节入口，默认展示 `top 3`，`top_k` 暂不纳入通用基线。
- 原因：这样既保留实验价值，又避免主界面被过多参数淹没，并保持 OpenAI-compatible 基线的稳定性。

## D-010：全局默认与会话覆盖规则定案

- 状态：已接受
- 决策：全局设置负责默认值，会话级配置负责单会话覆盖；当会话偏离默认值时，采用轻提示而不是打断式提醒。
- 原因：这能同时保证默认体验、实验灵活性，以及后续复盘时对实验条件的可感知性。

## D-011：`对话工作台` 布局定案

- 状态：已接受
- 决策：`对话工作台` 使用三栏布局；左栏负责会话导航，中栏负责聊天主流程，右栏负责参数与单轮请求观测；中栏顶部保留独立 `环境` 信息条，输入区下沿放置仅显示 `请求数` 与 `累计 Token` 的轻量状态读数；窄窗口场景下允许右栏折叠。
- 原因：这样既能保持聊天主流程顺畅，又能让实验环境与运行观测稳定可见，同时避免主工作台退化成普通聊天壳或过度膨胀成仪表盘。

## D-012：`会话分析` 页面指标定案

- 状态：已接受
- 决策：`会话分析` 页面以单会话复盘为目标；首屏总览展示 `请求总数 / 累计 Token / 累计成本估算 / 平均延迟`，其中成本明确为估算且需要解释支撑；趋势区以 `Token 趋势` 为主、`延迟趋势` 为辅、`成本趋势` 弱化；实验条件变化记录区首版保留，但仅在关键变化发生时低存在感呈现。
- 原因：这样既能保留对资源、体验和经济代价的核心感知，又不会把首版分析页做成噪音很大的数据后台。

## D-013：全局设置与会话覆盖交互定案

- 状态：已接受
- 决策：新会话创建时继承全局默认值；会话内参数修改只影响当前会话；全局设置应用按钮只影响未来新会话，已有会话保持不变；偏离默认值时采用三层提示；首版不做单参数恢复默认，但保留整场会话一键恢复默认环境，按钮位于右栏主提示区域并带轻确认。
- 原因：这样既能保持默认环境心智的稳定，又能为单会话实验保留足够灵活性，并避免全局设置误伤已有实验。

## D-014：请求生命周期与消息流定案

- 状态：已接受
- 决策：`RequestRecord` 在发请求前先创建；用户消息发送后立即写入；成功后再写助手消息；失败时保留请求记录和用户消息，但不把半截助手输出落成正式消息；失败场景允许 `UsageRecord` 部分存在；失败后可提供基于原请求条件的轻量重试动作，并为重试创建新的 `RequestRecord`。
- 原因：这样既能保证成功链路和失败链路的事实完整性，又能为可观测性、分析页和未来流式输出扩展保留稳定的数据基础。

## D-015：流式输出与失败恢复定案

- 状态：已接受
- 决策：流式文本在完成前只作为临时渲染存在，不额外显示文字型“生成中”标识，仅通过消息尾部光点或细小加载态表达未完成状态；首版支持手动停止生成，且生成中时发送按钮切换为中止按钮；中断、失败、手动停止都不把半截助手输出转为正式消息；失败状态主提示靠近消息区，输入区附近保留轻量重试承接。
- 原因：这样既能保留流式对话的即时反馈，又能清楚维护临时态与正式态的边界，并避免把首版失败恢复做成过重的故障系统。

## D-016：本地持久化与配置存储定案

- 状态：已接受
- 决策：长期事实落盘、临时流式态不落盘；用户消息立即保存、助手消息完成后保存；`pending` 请求立即落盘；未发送草稿只做轻量会话级暂存；`API Key` 采用系统级安全存储并与普通配置分层；删除会话时删除会话级明细，但允许通过更高层级的全局消耗账本保留已发生资源消耗事实。
- 原因：这样既能保证实验事实的完整性，又能避免把临时态误存成历史，同时为敏感凭据、失败请求和删除语义提供清晰边界。

## D-017：`v0.1` 实现优先级与里程碑定案

- 状态：已接受
- 决策：`v0.1` 分为 `M1 / M2 / M3` 三个实施里程碑；`M1` 打通消息、请求、存储与 `API Key` 安全存储主链；`M2` 完成实验台属性、基础可观测性、流式输出与手动停止生成；`M3` 再完成会话分析页与整体体验收口。会话分析页放在 `M3`，流式输出与手动停止生成放在 `M2`，`API Key` 的系统级安全存储属于 `M1` 阻塞项。
- 原因：这样能确保首版先把主链与可信边界站稳，再逐步叠加实验体验与分析能力，避免开发阶段边做边散。

## D-018：`M1` 开发任务分解定案

- 状态：已接受
- 决策：`M1` 拆为 `G1 ~ G6` 六组任务；允许先用非流式请求打通主链；`G2` 与 `G3` 可并行推进，但真实请求打通依赖凭据主链可用；`G5` 从一开始就接真实存储适配层；`G6` 在 `M1` 只要求“最近一次请求”观测，不做历史观测面板。
- 原因：这样能把 `M1` 收束为一条最小但可信的实现闭环，先验证消息、请求、持久化与凭据边界，再为后续 `M2 / M3` 叠加体验与分析能力。

## D-019：`v0.1` 实现基线与工程选型定案

- 状态：已接受
- 决策：`v0.1` 正式采用 `Tauri 2 + React + TypeScript + Vite` 作为桌面与前端主栈；状态层采用 `Zustand`，并继续按 `领域 Store + Service + Repository` 分层；`SQLite` 作为 `M1` 正式事实库；`API Key` 通过 `Tauri` 侧系统级安全隔离能力管理；价格表在 `M1` 采用内置静态配置，并保留隐藏的高级覆盖入口；应用重启后在 `Zustand rehydrate` 阶段清洗遗留运行态，凡是 `streaming` 或 `sent` 的消息状态都视为僵尸状态并立即收尾为 `interrupted` 或 `error` 等失败语义。
- 原因：这样既能把桌面优先、本地优先、敏感凭据分层、真实持久化与可观测性要求收束成一条可执行工程路线，又能避免应用异常退出后残留“仍在生成中”的伪状态，保证 `M1` 的主链闭环可信。
