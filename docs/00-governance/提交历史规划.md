# 提交历史规划

## 目的

这份文档不规定 Git 教条，而是为了让 `Ego` 的提交历史尽量具备这几个特性：

- 能从提交标题直接看出当前阶段在推进什么
- 能把“规范阶段”和“实现阶段”区分开
- 能让后续回看 `M1 / M2 / M3` 时，不需要重新猜这次改动的边界

## 当前基线

仓库当前已经有首个基线提交：

- `chore: bootstrap ego v0.1 foundation`

它的作用是把 `v0.1` 的文档基线、前端骨架、Tauri 桌面壳、本地存储、流式主链基础一起落到仓库里。

这个提交已经比较大，后续应避免继续堆积同体量的“全打包式提交”。

## 后续提交原则

后续提交尽量遵循以下原则：

1. 一个提交只解决一类主问题  
   例如：流式输出、停止生成、失败重试、默认值同步、UI 收口，不要混在同一次提交里。

2. 优先按“主链能力”切提交  
   比起按文件切，更推荐按用户可感知能力切：
   - `feat: add streaming response pipeline`
   - `feat: support cancel during streaming`
   - `feat: retry latest failed request`

3. 文档与实现可以分提交  
   如果某一轮包含：
   - 先补 Spec
   - 再落代码  
   那文档和实现可以拆成两次提交，避免回看时混淆。

4. UI 收口优先走 `fix` 或 `refactor`，不要伪装成大功能  
   例如：
   - `fix: tighten workbench layout and bottom composer behavior`
   - `refactor: simplify conversation message presentation`

## 建议提交前缀

- `chore:` 仓库基础设施、忽略规则、依赖调整、脚手架整理
- `docs:` 文档补充、规范收口、README 更新
- `feat:` 新能力、新链路、新交互
- `fix:` 缺陷修复、状态错误、同步问题、边界问题
- `refactor:` 不改变行为的结构性整理

## 推荐顺序

如果继续沿当前节奏推进，推荐后续提交历史尽量靠近下面这个顺序：

1. `docs: add commit plan and refresh local startup guide`
2. `feat: add streaming response pipeline for conversation requests`
3. `feat: support cancel during active streaming`
4. `feat: retry latest failed request from conversation workspace`
5. `fix: sync inherited conversation settings after default updates`
6. `fix: tighten workbench layout and observability density`

## 当前阶段建议

当前最值得保持的习惯是：

- `M2` 的每个主能力单独提交
- 先让一条链路完整闭环，再进入下一条
- 不把“顺手修的 UI、小文案、状态同步”无限混进主能力提交

大白话说：

先把每一刀切干净，  
后面你回头看，仓库自己就会讲故事。
