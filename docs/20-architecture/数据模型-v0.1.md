# Ego `v0.1` 数据模型

## 1. 建模目标

当前数据模型要服务于三件事：

- 对话
- 记忆实验
- 可观测性

同时要刻意避免过早把 Agent、工作流之类更复杂的对象揉进来。

## 2. 核心实体

### 2.1 `Conversation`

表示一个聊天工作区。

建议结构：

```ts
type Conversation = {
  id: string
  title: string
  createdAt: string
  updatedAt: string
  modelConfigId: string
  memoryPolicyId: string
  archived?: boolean
}
```

### 2.2 `Message`

表示会话中的一条消息。

建议结构：

```ts
type Message = {
  id: string
  conversationId: string
  role: 'system' | 'user' | 'assistant'
  content: string
  createdAt: string
  requestId?: string | null
}
```

### 2.3 `ModelConfig`

表示一个会话或一次请求所使用的模型配置。

建议结构：

```ts
type ModelConfig = {
  id: string
  providerLabel: string
  baseUrl: string
  model: string
  temperature: number
  maxCompletionTokens?: number | null
  createdAt: string
  updatedAt: string
}
```

### 2.4 `MemoryPolicy`

定义下一次请求发出前，如何选择过去的对话内容。

建议结构：

```ts
type MemoryPolicy = {
  id: string
  type: 'none' | 'recent-window' | 'summary-plus-recent'
  recentTurnCount?: number | null
  summaryPromptTemplate?: string | null
  updatedAt: string
}
```

大白话解释：
这个对象本质上就是一份“带哪些旧聊天内容一起发给模型”的规则说明书。

### 2.5 `RequestRecord`

表示一次真正发出去的模型请求。

建议结构：

```ts
type RequestRecord = {
  id: string
  conversationId: string
  modelConfigSnapshot: {
    providerLabel: string
    baseUrl: string
    model: string
    temperature: number
    maxCompletionTokens?: number | null
  }
  memoryPolicySnapshot: {
    type: 'none' | 'recent-window' | 'summary-plus-recent'
    recentTurnCount?: number | null
  }
  promptMessageId: string
  startedAt: string
  finishedAt?: string | null
  status: 'pending' | 'completed' | 'failed'
  errorMessage?: string | null
}
```

这里为什么要有快照：
因为用户后面可能会修改配置，但历史请求记录必须保留“当时真实发生了什么”。

### 2.6 `UsageRecord`

表示一次请求的 usage 数据，无论它来自提供方还是本地估算。

建议结构：

```ts
type UsageRecord = {
  id: string
  requestId: string
  inputTokens?: number | null
  outputTokens?: number | null
  totalTokens?: number | null
  latencyMs?: number | null
  estimatedCostUsd?: number | null
  usageSource: 'provider-reported' | 'locally-estimated' | 'unknown'
  createdAt: string
}
```

### 2.7 `ConversationSummary`

只有在摘要型记忆策略开启时才需要。

建议结构：

```ts
type ConversationSummary = {
  id: string
  conversationId: string
  content: string
  sourceMessageCutoffId: string
  createdAt: string
  updatedAt: string
}
```

## 3. 关系概览

- 一个 `Conversation` 包含多条 `Message`
- 一个 `Conversation` 包含多条 `RequestRecord`
- 一个 `RequestRecord` 通常对应一条 `UsageRecord`
- 一个 `Conversation` 可以有零条或一条当前生效的 `ConversationSummary`
- 一个 `Conversation` 关联一个当前生效的 `ModelConfig`
- 一个 `Conversation` 关联一个当前生效的 `MemoryPolicy`

## 4. 有意不纳入本期的实体

下面这些对象当前明确不进入 `v0.1`：

- `KnowledgeBlock`
- `AgentRun`
- `WorkflowRun`
- `ToolCall`
- `PermissionGrant`

这不是漏设计，而是有意做范围控制。

## 5. 数据设计说明

### 5.1 `Message` 和 `RequestRecord` 分开

`Message` 描述用户看见的对话内容。
`RequestRecord` 描述底层运行时真正发生的一次请求事件。

大白话解释：
一个是“用户看到的聊天内容”，一个是“系统背后干了什么”。这两者不应该混成一个对象。

### 5.2 `UsageRecord` 和请求元数据分开

usage 数据可能缺失、部分返回、延迟返回，或者只能估算。

把它单独拆开，可以避免让请求对象承担太多不稳定字段。

### 5.3 保留历史真实状态

即使用户后来改了模型配置和记忆策略，历史请求也应该保留当时的配置快照。

## 6. 留待后续批准的问题

- `system` Prompt 是全局配置、会话级配置，还是请求级覆盖
- Token 估算值是永久存储，还是按需重新计算
- 价格表是让用户自行配置，还是在实验期先写死
