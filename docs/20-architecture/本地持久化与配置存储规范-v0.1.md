# Ego `v0.1` 本地持久化与配置存储规范

## 1. 文档定位

本文件定义 `v0.1` 中哪些数据应持久化到本地、哪些只属于运行时状态，以及配置与记录的存储边界。

它解决的核心问题是：

- 哪些是长期事实
- 哪些只是当前过程
- 哪些属于普通配置
- 哪些属于敏感凭据

大白话解释：
这份规范解决的是“什么应该被长期保存，什么只该短暂存在”，避免把临时态误存成历史，或者把历史事实只留在内存里。

## 2. 核心目标

这份规范需要同时解决五件事：

1. 明确哪些对象必须持久化
2. 明确哪些状态只属于运行时
3. 规定配置数据和业务记录的分界
4. 说明失败与取消场景的数据保存边界
5. 为首版实现控制复杂度

## 3. 核心原则

### 3.1 长期事实要落盘，临时过程不强留

凡是用户后续会依赖、会复盘、会再次查看的事实，应持久化。

凡是只服务当前交互过程、完成后就失去意义的临时态，不应默认落盘。

### 3.2 用户明确产生的数据优先保存

用户明确发出的输入、用户主动配置的默认值、已完成或已失败的请求记录，都属于应优先保存的数据。

### 3.3 未完成态默认不进入长期历史

例如：

- 正在流式生成中的临时文本
- 尚未完成的半截助手输出
- 仅用于动画和状态反馈的 UI 标志

这些不应被直接持久化为长期事实。

### 3.4 配置与运行记录分开

全局设置、会话覆盖值、消息内容、请求记录、usage 记录，本质上不是一类数据。

因此在结构上应明确区分，而不是混成一个不透明大对象。

## 4. 建议持久化的对象

根据当前 `v0.1` 范围，以下对象必须持久化：

- `Conversation`
- `Message`
- `ModelConfig`
- `MemoryPolicy`
- `RequestRecord`
- `UsageRecord`

### 4.1 `Conversation`

应持久化。

原因：

- 它是会话入口
- 它承载后续所有消息和请求的归属

### 4.2 `Message`

应持久化，但要区分：

- 用户消息：发送后立即持久化
- 助手消息：仅在完整完成后持久化

不应持久化：

- 流式中尚未完成的半截助手临时文本

### 4.3 `ModelConfig`

应持久化。

原因：

- 它代表全局默认环境
- 也可能用于会话级覆盖的来源基础

### 4.4 `MemoryPolicy`

应持久化。

原因：

- 它是实验条件的一部分
- 后续会影响复盘解释

### 4.5 `RequestRecord`

应持久化，包括：

- `pending` 创建
- `completed`
- `failed`
- `cancelled/aborted`

正式结论：

- `pending` 状态的 `RequestRecord` 应立即落盘
- 它属于已经发生、但尚未结束的请求事实
- 首版允许本地存在未完成请求记录
- 应用重启后，不尝试继续恢复旧的未完成生成
- 在 `Zustand persist` 状态恢复的 `rehydrate` 钩子中，必须清洗遗留运行态
- 只要发现消息运行状态为 `streaming` 或 `sent`，就视为异常退出留下的僵尸状态
- `streaming` 必须立刻收尾为 `interrupted`
- `sent` 必须立刻收尾为 `error` 或等价失败态
- 清洗后，界面中不允许继续残留“仍在生成中”的假状态

### 4.6 `UsageRecord`

应持久化，但允许部分字段缺失。

原因：

- usage 是核心观测对象
- 即使失败场景也可能有部分真实消耗

## 5. 建议只保留在运行时的状态

以下内容建议只存在于运行时，不默认持久化：

- 当前是否正在流式生成
- 尾部光点/细小加载态
- 流式中临时渲染的半截助手文本
- `streaming / sent / interrupted / error` 这类 Store 侧消息运行状态
- 当前 UI 是否展开某个折叠区
- 当前 hover 或悬浮提示状态
- 当前输入框中的未发送草稿（不进入正式历史，但可做轻量会话级暂存）

大白话解释：
这些状态的意义主要是让界面“当下更顺”，不是为了让它们成为历史事实。

补充说明：

- 上述运行态不进入 `SQLite` 正式历史
- 但允许随 `Zustand persist` 轻量快照短暂保留，以支持重启后的统一清洗与收尾

## 6. 全局配置与会话覆盖的存储边界

### 6.1 全局配置

应持久化：

- API Base URL
- 默认模型
- 默认 `temperature`
- 默认 `top_p`
- 默认 `max_output_tokens`
- 默认 `memory_policy`
- 价格表配置

这些属于：

- 全局默认环境
- 后续新会话的起跑线

补充约束：

- `API Key` 不按普通全局配置处理
- 它属于敏感凭据，应与普通偏好设置分层存储

### 6.2 敏感凭据存储

`API Key` 这类敏感凭据在 `v0.1` 中采用系统级安全存储方案。

正式结论：

- `API Key` 不与普通配置混存
- 优先使用操作系统提供的安全凭据存储能力
- 普通配置仍保留在本地配置层

大白话解释：
`API Key` 更像“本地凭据”，不是“本地偏好设置”。

### 6.3 会话覆盖值

应持久化：

- 当前会话实际使用的 `model`
- 当前会话实际使用的 `temperature`
- 当前会话实际使用的 `top_p`
- 当前会话实际使用的 `max_output_tokens`
- 当前会话实际使用的 `memory_policy`

原因：

- 它们决定了当前会话真实跑在什么实验环境里
- 后续分析必须依赖这些事实

## 7. 草稿存储边界

未发送草稿不作为正式长期历史保存。

正式结论：

- 支持轻量会话级草稿暂存
- 草稿只服务恢复编辑
- 不进入 `Message`
- 不进入 `RequestRecord`
- 不进入分析页
- 不进入正式历史体系

大白话解释：
草稿可以被“找回来”，但不应该被“当成历史事实保存”。

## 8. 失败与取消场景的保存边界

### 8.1 失败请求

应保存：

- 用户消息
- `RequestRecord(status=failed)`
- 可得的部分 `UsageRecord`

不应保存：

- 半截助手正式消息

### 8.2 用户手动中止

应保存：

- 用户消息
- `RequestRecord(status=cancelled/aborted)`
- 可得的部分 usage 或耗时

不应保存：

- 半截助手正式消息

## 9. 会话删除语义

这是首版最容易混乱的一块之一。

正式结论：

- 删除会话时，不保留悬空的 `UsageRecord`
- 会话级的 `Message / RequestRecord / UsageRecord` 明细可随会话一并删除
- 但“已发生的资源消耗”允许通过更高层级的全局消耗账本保留

大白话解释：
删除会话时，细粒度实验轨迹可以消失，但系统仍可以保留更高层级的总消耗事实，而不是留下无主的 usage 明细。

## 10. 何时落盘

### 10.1 立即落盘型

这类数据一旦发生，应尽快落盘：

- 用户消息
- 新建会话
- 全局设置变更
- 会话覆盖值变更
- `RequestRecord` 创建与状态更新

原因：

- 这些都属于不能轻易丢失的事实

### 10.2 完成后落盘型

这类数据应在形成稳定事实后再落盘：

- 助手正式消息
- 最终 usage 补齐信息

## 11. 首版不建议做重的存储能力

以下内容不建议在 `v0.1` 首版做太重：

- 自动本地草稿恢复系统之外的复杂草稿管理
- 多版本快照回放
- 复杂存储迁移工具链
- 临时流式片段缓存系统
- 失败恢复队列持久化

原因：

- 会显著增加实现复杂度
- 会让存储边界过早复杂化

## 12. 当前定案结论

`v0.1` 当前正式结论如下：

- 长期事实落盘，临时流式态不落盘
- 用户消息立即保存，助手消息完成后保存
- 失败和取消请求都保留记录
- `UsageRecord` 允许部分存在
- `pending` 请求立即落盘
- 未发送草稿支持轻量会话级暂存，但不进入正式历史体系
- `API Key` 采用系统级安全存储方案，不与普通配置混存
- 配置与运行记录保持分层存储
- 应用重启后通过 `rehydrate` 清洗遗留的 `streaming / sent` 僵尸运行态
- 删除会话时删除会话级明细，但允许保留更高层级的全局消耗账本
