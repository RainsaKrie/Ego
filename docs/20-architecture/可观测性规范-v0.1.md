# Ego `v0.1` 可观测性规范

## 1. 为什么这件事重要

可观测性不是一个隐藏在后台的工程调试能力，而是产品本身的一部分。

`v0.1` 的目标不只是“能聊天”，而是让我们真正理解每次请求到底花了什么、为什么会花这么多。

## 2. 核心指标

### 2.1 Token 使用量

在可能的情况下，系统应追踪：

- 输入 Token
- 输出 Token
- 总 Token

大白话解释：
输入 Token 是你发给模型的内容，输出 Token 是模型回给你的内容，总 Token 则直接影响成本和上下文膨胀速度。

### 2.2 延迟

系统应追踪：

- 请求开始时间
- 如果后续支持的话，首个响应到达时间
- 完整返回结束时间

对 `v0.1` 而言，至少要保证“完整完成延迟”可观测。

### 2.3 成本估算

系统应追踪：

- 当前采用的价格依据
- 单次请求成本估算
- 会话累计成本估算

### 2.4 上下文体积

系统应追踪：

- 本次真正被打包进请求的消息数
- 打包后的上下文近似规模
- 当前生效的记忆策略

这样做是为了帮助解释：为什么看起来差不多的两个 Prompt，成本却可能差很多。

## 3. 数据来源等级

每一条 usage 数据都必须声明来源：

- `provider-reported`
- `locally-estimated`
- `unknown`

原因：
并不是所有提供方都会稳定返回 usage 信息，尤其在流式模式下更容易缺失。

## 4. 请求级观测记录

每次请求的分析视图应尽量展示：

- 时间戳
- 模型
- `temperature`
- 记忆策略
- 延迟
- Token 使用量
- 成本估算
- usage 数据来源可信度
- 是否包含摘要记忆

## 5. 会话级聚合

在会话层面，系统应能汇总：

- 请求总数
- 总 Token 消耗
- 平均延迟
- 累计成本估算
- 平均每轮 Token

## 6. 比较目标

这套可观测性结构未来应该支持回答类似问题：

- 相同 Prompt，不同 `temperature` 有什么差异
- 相同会话，不同记忆策略有什么差异
- 相同任务，不同模型的成本和效果差异是什么

`v0.1` 不需要一步做成完整实验仪表盘，但存储结构必须为后续比较能力预留空间。

## 7. 估算策略

当提供方没有返回 usage 数据时，允许使用本地估算。

关键规则：
UI 必须明确标记哪些值是估算值。

大白话解释：
估算值依然有参考意义，但它不能伪装成精确账单。

## 8. 定价策略

对于 `v0.1`，成本应被视为基于当前模型价格表的估算值。

这意味着后续架构最好支持：

- 模型级价格配置
- 输入和输出分开计价
- 随时间变化的价格版本

## 9. 失败场景

可观测性必须能承受部分失败：

- 如果响应成功但 usage 缺失，仍然要存请求记录
- 如果请求失败，仍然要存失败信息和已知耗时
- 如果价格不可用，就只存 usage，不要因为缺少成本而阻塞整条记录

## 10. 产品纪律

可观测性的目标是帮助用户回答“刚刚到底发生了什么”，而不是逼着用户去读原始 JSON。

所以这些数值必须以结构化方式被保存，而不是只打进日志或控制台。
