# Ego `v0.1` 架构基线

## 1. 架构定位

Ego `v0.1` 是一个桌面优先、本地优先的 AI 对话实验台。

它现在还不是完整意义上的 AI 操作系统运行时，而是未来可以逐步长成 AI OS 的最小可用切片。

## 2. 核心架构原则

### 2.1 本地优先

用户对话、运行配置和可观测性数据，默认都应以本地存储为主。

大白话解释：
这意味着应用应该把“本机”当成用户数据的真正家，而不是把它当成未来上云前的临时缓存。

### 2.2 `BYOK` 边界

系统默认假设用户自己提供模型凭据。

这样做能让早期架构更轻，也能让成本和 Token 观测更透明。

### 2.3 可观测性是产品价值，不是调试副产物

请求元数据不是只给开发者看的调试废料，而是产品核心价值的一部分。

这意味着 usage、延迟、成本、记忆策略和请求形态，都应该被认真建模，而不是只埋在日志里。

### 2.4 记忆范围要收窄

`v0.1` 只处理单个聊天会话内部的对话记忆策略。

它不尝试在当前阶段解决长期个人知识记忆问题。

### 2.5 优雅降级

如果提供方没有返回完整 usage 数据，应用依然必须可用，并明确告诉用户哪些值是精确值，哪些是估算值。

## 3. 建议的系统分层

### 3.1 工作台壳层

职责：

- 应用窗口壳层
- 导航结构
- 设置入口
- 会话切换

### 3.2 对话运行时

职责：

- 发送模型请求
- 接收模型响应
- 管理会话消息历史
- 在每次请求前应用记忆策略

### 3.3 可观测性层

职责：

- 追踪每次请求的 usage 和延迟
- 存储成本估算
- 提供聚合与比较能力

### 3.4 持久化层

职责：

- 本地保存会话
- 本地保存模型设置
- 本地保存 usage 记录和摘要结果

### 3.5 提供方适配层

职责：

- 把内部请求规范化为 OpenAI 兼容接口请求
- 规范化提供方响应
- 尽可能提取 usage 字段

大白话解释：
这一层本质上像一个翻译器。它的作用是让应用可以对接不同兼容接口，而不让上层业务到处关心每家返回格式的小差别。

## 4. 建议的模块边界

为了后续实现清晰，建议先按下面的模块思路规划：

- `app-shell`
- `conversation`
- `memory`
- `provider`
- `observability`
- `settings`
- `storage`

这样以后即使接入知识检索或工具调用，也不会一开始就把所有逻辑搅在一起。

## 5. 请求生命周期

建议的高层流程如下：

1. 用户选择一个会话并输入 Prompt。
2. 系统读取当前生效的模型配置和记忆策略。
3. 系统构建本次请求真正要发送的上下文。
4. 系统通过提供方适配层发起规范化请求。
5. 系统记录时间边界。
6. 系统存储助手回复。
7. 系统存储精确或估算的 usage 元数据。
8. 系统更新会话级统计结果。

## 6. 为什么暂缓 Agent

`v0.1` 明确不做 Agent，不是因为它不重要，而是因为它过早。

原因在于：

- Agent 需要任务生命周期管理
- Agent 需要工具权限边界
- Agent 需要失败重试和恢复机制
- Agent 需要更复杂的执行日志模型

大白话解释：
聊天大体上还是“一问一答”的模式，而 Agent 更像一个小型运行平台。它不适合被硬塞进聊天 MVP。

## 7. 面向未来的兼容约束

虽然 `v0.1` 范围很窄，但当前设计仍应避免堵死未来扩展到以下能力的路：

- 知识检索
- 长期记忆
- 工具调用
- Agent 执行
- 工作流编排

为了保留这些扩展空间，对话数据和可观测性数据不应该被揉成一个不透明的大对象。

## 8. 当前推荐的技术方向

从架构层面看，`v0.1` 当前应优先偏向：

- 桌面优先壳层
- 具备扩展空间的本地持久化
- 先以 OpenAI 兼容接口为统一契约
- 显式、简单的记忆策略
- 每次请求独立记录的可观测性结构

更细的技术选型，可以在进入实现阶段前继续逐项评审并批准。
