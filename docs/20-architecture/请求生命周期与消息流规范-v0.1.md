# Ego `v0.1` 请求生命周期与消息流规范

## 1. 文档定位

本文件定义 `v0.1` 中一次对话请求从发起到落地的完整生命周期，以及消息、请求记录、usage 记录之间的关系。

它解决的核心问题是：

`用户点一次发送，到底发生了什么？`

大白话解释：
这份规范是在给“发送一条消息”这件事画出真实流程图，避免后面实现时每个人脑子里的流程都不一样。

## 2. 核心目标

这份规范需要同时解决五件事：

1. 定义一次请求的标准阶段
2. 明确消息、请求记录、usage 记录各自何时创建
3. 说明 UI 在每个阶段应该看到什么状态
4. 说明请求失败时如何降级
5. 为流式输出和未来扩展保留空间

## 3. 核心对象回顾

根据现有数据模型，这条链路至少涉及：

- `Conversation`
- `Message`
- `RequestRecord`
- `UsageRecord`

它们分别承担：

- `Conversation`：承载整个会话
- `Message`：承载用户和助手看到的对话内容
- `RequestRecord`：承载一次真实发出的模型请求事件
- `UsageRecord`：承载 token、延迟、成本等观测数据

大白话解释：
`Message` 是聊天内容，`RequestRecord` 是后台事件，`UsageRecord` 是运行账单。

## 4. 标准请求生命周期

`v0.1` 把一次请求拆成 7 个阶段。

### 4.1 阶段一：用户准备发送

触发点：

- 用户在当前会话输入 Prompt
- 用户点击发送

此时系统需要做的事：

- 读取当前会话参数
- 读取当前记忆策略
- 读取当前会话历史

这个阶段还没有真正发请求。

### 4.2 阶段二：构建有效请求上下文

系统根据当前规则构建本次真正要发给模型的内容，包括：

- 当前用户输入
- 当前系统 Prompt（如果有）
- 根据记忆策略选择的历史消息
- 当前会话生效参数

大白话解释：
用户看到的是整段聊天，但真正发出去的，不一定是全部历史，而是按记忆策略打包后的有效上下文。

### 4.3 阶段三：创建请求记录

在真正发请求之前，系统应先创建一条 `RequestRecord`。

建议至少记录：

- `conversationId`
- 当前参数快照
- 当前记忆策略快照
- `status = pending`
- `startedAt`

正式结论：

- `RequestRecord` 必须在发请求前就先创建
- 初始状态为 `pending`
- 请求结束后再更新为 `completed` 或 `failed`

原因：

- 即使请求失败，也能留下“这次尝试过”的运行痕迹
- usage、延迟、错误信息需要稳定归属点

### 4.4 阶段四：落地用户消息

用户点击发送后，用户消息应立即进入消息流，并写入 `Message`。

正式结论：

- 用户消息在发送后立即写入
- 不等待模型成功再写
- 请求失败时，用户消息仍然保留

原因：

- 这是用户已经明确提交的输入
- 即使模型失败，这条用户消息也是真实发生过的事实

### 4.5 阶段五：请求执行与响应接收

系统通过 provider adapter 发起请求，并接收响应。

这里需要兼容两种未来形态：

- 非流式返回
- 流式返回

对 `v0.1` 来说，即使首版不马上做复杂流式，也应在生命周期定义上给它留口子。

### 4.6 阶段六：落地助手消息

当模型响应完成后，系统创建并保存助手 `Message`。

这一阶段应保证：

- 用户在消息流中看到最终助手内容
- 助手消息与对应请求可以建立关联

失败场景下的正式结论：

- 首版失败时，不保留半截助手输出为正式消息
- 流式中途内容可以临时渲染
- 最终失败则不落正式助手 `Message`

大白话解释：
“看见过”不等于“应该作为正式回复存档”。

### 4.7 阶段七：写入 usage 与收尾

请求结束后，系统应补齐：

- `RequestRecord.status`
- `finishedAt`
- `errorMessage`（如失败）
- `UsageRecord`

如果提供方返回 usage：

- 标记为 `provider-reported`

如果提供方未返回 usage，但本地能估算：

- 标记为 `locally-estimated`

如果两者都拿不到：

- 标记为 `unknown`

正式结论：

- 失败场景允许 `UsageRecord` 部分存在
- 允许字段缺失
- 允许只记录耗时
- 允许只记录估算值
- 必须明确区分完整度和来源

## 5. 成功链路

一次成功请求的推荐顺序如下：

1. 用户点击发送
2. 系统构建有效上下文
3. 创建 `RequestRecord(pending)`
4. 写入用户 `Message`
5. 发起请求
6. 接收响应
7. 写入助手 `Message`
8. 写入 `UsageRecord`
9. 更新 `RequestRecord(completed)`

## 6. 失败链路

请求失败时，推荐规则如下。

### 6.1 用户消息保留

用户消息已经明确发送，因此应保留。

### 6.2 请求记录保留

失败并不意味着这次尝试不存在。

应保留：

- `RequestRecord`
- `status = failed`
- 错误信息
- 已知耗时（如果可得）

### 6.3 助手消息处理

如果模型完全没有产生有效内容：

- 不写入正式助手消息

如果未来存在部分流式内容但最终失败：

- 不把半截输出当成正式稳定消息

### 6.4 usage 处理

如果失败时仍然能拿到部分 usage 或耗时信息：

- 允许记录部分 `UsageRecord`

如果拿不到：

- 仍然保留失败的 `RequestRecord`
- 不强行补伪 usage

## 7. UI 状态建议

建议至少有以下状态：

### 7.1 发送中

当请求发出后，UI 应明确告诉用户当前正在等待模型响应。

### 7.2 响应完成

助手消息完整落地，usage 和请求状态补齐。

### 7.3 请求失败

用户应知道这次失败了，但用户消息仍然保留。

补充建议：

- 失败后可提供一次轻量“重试/刷新”动作
- 该动作更适合做成轻量图标按钮，而不是长文案按钮
- 详细语义可放在悬浮提示中，例如“重试”或“按原条件重新生成”

### 7.4 usage 不完整

如果响应成功，但 usage 只有估算值或完全未知，UI 应明确区分，而不是伪装成精确账单。

## 8. 失败重试原则

若请求失败，允许用户发起一次轻量重试。

正式结论：

- 重试按钮应轻量化
- 更适合使用图标或极短文案
- 详细语义不堆在主界面，而放到悬浮提示中
- 重试应基于原请求条件重新生成
- 新重试请求应生成新的 `RequestRecord`
- 不能覆盖旧失败记录

大白话解释：
失败重试不是“把半截内容补完”，而是“按原条件重新来一次”。

## 9. 流式输出的预留原则

即使 `v0.1` 首版不优先复杂实现流式，也建议规范先预留以下原则：

- 流式内容是“响应接收中的一种形式”
- 最终正式入库的助手消息，应以完成后的最终内容为准
- 不能因为流式中途失败，就把半截内容默认当成正式稳定记录

这不是现在就要求全部实现，而是避免以后加流式时推翻数据链路。

## 10. 当前定案结论

`v0.1` 当前正式结论如下：

- `RequestRecord` 先建，再发请求
- 用户消息发送后立即写入
- 成功后再写助手消息
- 失败时保留请求记录和用户消息
- 失败时不把半截助手输出落成正式消息
- `UsageRecord` 尽量记录，但允许部分缺失
- UI 必须能区分完整 usage、估算 usage 与未知 usage
- 失败后可提供轻量重试动作
- 重试基于原请求条件重新生成，并创建新的 `RequestRecord`
