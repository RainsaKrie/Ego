# Ego `v0.1` 参数分层与配置规则

## 1. 文档定位

本文件定义 `v0.1` 的参数分层方式，以及全局默认配置与会话级覆盖的边界。

目标不是把参数做得越多越好，而是保证：

- 参数足够支撑实验
- 界面不会变成一面难以理解的控制台
- 不同层级的配置职责清晰

## 2. 参数分层原则

### 2.1 第一层：高频实验参数

这类参数应直接出现在 `对话工作台` 的右侧常驻区。

它们的特征是：

- 调整频率高
- 对输出风格和成本影响直接
- 对 `v0.1` 的学习目标价值很高

纳入这一层：

- `model`
- `temperature`
- `top_p`
- `max_output_tokens`
- `memory_policy`

### 2.2 第二层：高级参数

这类参数应放入折叠区，不默认全部展开。

它们通常满足：

- 不是每轮都需要修改
- 对实验有帮助，但不应干扰主流程
- 更适合在掌握基础参数后再使用

纳入这一层：

- `seed`
- `truncation`

说明：

- `seed` 用于帮助实验复现，不是用来让模型“更聪明”
- `truncation` 更偏请求边界控制，不属于日常风格调节参数

### 2.3 第三层：高级观测项

这类内容不是主要的控制旋钮，更像分析仪器。

它们不应进入主参数区，但可以作为高级观测能力存在。

纳入这一层：

- `top_logprobs`

大白话解释：
它帮助我们看清模型当时“差点还会怎么答”，更像显微镜，不像方向盘。

### 2.4 第四层：提供方专属参数

这类参数不进入 `v0.1` 的通用主界面。

原因通常是：

- 并不是所有 OpenAI 兼容接口都支持
- 不适合作为跨提供方统一基线
- 过早暴露会制造学习噪音

代表项：

- `top_k`
- 提供方私有 penalty 参数
- 某些本地模型专属采样参数

## 3. 为什么 `top_p` 纳入而 `top_k` 暂不纳入

### 3.1 `top_p`

`top_p` 纳入 `v0.1` 通用基线。

原因：

- 它是常见的采样控制参数
- 与 `temperature` 一起，足够支撑早期采样机制实验
- 在 OpenAI 当前官方接口语义中属于明确存在的主参数

大白话解释：
`top_p` 像是先把很不靠谱的一大批候选词排除掉，再在前面那一小群更可能的词里选。

### 3.2 `top_k`

`top_k` 暂不纳入 `v0.1` 通用基线。

原因：

- 它更常见于开源模型和部分本地推理框架
- 对 OpenAI-compatible 统一基线来说不够稳定
- 过早暴露会增加参数学习负担

大白话解释：
`top_k` 的逻辑是“每一步只允许在最可能的前 K 个词里选”，它当然有价值，但不适合作为 `v0.1` 的标准主旋钮。

## 4. `seed` 的规则

`seed` 进入 `v0.1`，但不作为常驻主参数。

规则：

- 位于高级参数折叠区
- 默认不启用
- 请求记录中应保留该值，便于复盘和复现

定位：
它不是提高质量的旋钮，而是帮助做控制变量实验的工具。

## 5. `top_logprobs` 的规则

`top_logprobs` 在 `v0.1` 中显示，但不提供调节入口。

规则：

- 不作为用户可调参数
- 默认展示 `top 3`
- 优先放在单轮请求详情或分析页折叠区
- 若提供方不支持，则优雅降级为不显示

大白话解释：
它的价值在于观察模型“当时还想选什么词”，而不是直接改变模型输出。

## 6. 全局设置与会话级覆盖

### 6.1 全局设置

全局设置是整个应用的默认起跑线。

作用：

- 给新会话提供默认参数
- 定义整个应用的基础环境
- 减少重复配置

建议属于全局设置的内容：

- `api_base_url`
- `api_key`
- 默认 `model`
- 默认 `temperature`
- 默认 `top_p`
- 默认 `max_output_tokens`
- 默认 `memory_policy`
- 价格表配置

### 6.2 会话级覆盖

会话级覆盖是“当前实验会话”对默认值的局部改写。

作用：

- 允许单个会话偏离默认值
- 不影响其他会话
- 支持并行实验

建议支持会话级覆盖的内容：

- 当前会话 `model`
- 当前会话 `temperature`
- 当前会话 `top_p`
- 当前会话 `max_output_tokens`
- 当前会话 `memory_policy`

### 6.3 推荐行为

推荐采用以下规则：

1. 用户先在全局设置中定义默认值
2. 新建会话时自动继承这些默认值
3. 用户可在当前会话中单独覆盖
4. 覆盖只影响当前会话，不反向修改全局默认值

大白话解释：
全局设置负责默认起跑线，会话级覆盖负责这次实验单独怎么跑。

## 7. 会话级覆盖提示

当会话配置偏离全局默认值时，应给出轻提示。

建议：

- 使用轻量标签，而不是弹窗警告
- 标签点击后可查看偏离项明细
- 首版至少能看出“当前会话已偏离默认值”

原则：
这个提示不是为了阻止修改，而是为了保证实验条件可感知。

## 8. 主工作台中的参数展示建议

### 8.1 常驻显示

建议在主工作台右侧常驻显示：

- 当前 `model`
- 当前 `temperature`
- 当前 `top_p`
- 当前 `max_output_tokens`
- 当前 `memory_policy`

### 8.2 折叠显示

建议放入高级参数折叠区：

- `seed`
- `truncation`

### 8.3 请求详情或分析折叠区展示

建议放入请求详情或分析页折叠区：

- `top_logprobs` 的 `top 3` 候选结果

## 9. 当前定案结论

`v0.1` 当前正式结论如下：

- 采用分层参数结构
- 高频参数直接暴露在主工作台
- 高级参数进入折叠区
- `top_p` 纳入 `v0.1`
- `top_k` 暂不进入 `v0.1` 通用基线
- `seed` 进入高级参数层
- `top_logprobs` 显示但不开放调节，默认 `top 3`
- 全局设置负责默认值
- 会话级配置负责单会话覆盖
- 会话偏离默认值时使用轻提示
