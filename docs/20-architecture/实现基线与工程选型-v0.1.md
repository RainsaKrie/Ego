# Ego `v0.1` 实现基线与工程选型

## 1. 文档定位

本文件基于已经批准的产品与架构规范，给出进入 `M1` 实现阶段时的正式工程基线。

它解决的核心问题是：

- `M1` 采用什么桌面与前端主栈
- 状态、存储、凭据与 Provider 适配分别落在哪里
- 哪些工程约束已经正式拍板，不再停留在草案层

大白话解释：
这份文档是在把“已经讲清楚的产品和架构方向”，收束成“可以正式开工的工程路线”。

## 2. 现有约束复述

根据当前已经生效的文档，`v0.1` 的实现必须同时满足：

- 桌面优先、本地优先
- `BYOK`
- `M1` 先打通非流式主链
- `API Key` 不与普通配置混存
- 会话、消息、请求、usage 需要真实落盘
- 可观测性属于产品能力，不是调试副产物

因此，当前工程方案明确不走以下路线：

- 先做纯 Web，再整体补桌面壳层
- 先做纯内存，再整体切换持久化
- 把凭据塞进普通配置文件或数据库
- 只先做聊天 UI，再回头补运行时事实模型

## 3. 正式选型目标

当前工程选型优先服务以下目标：

1. 尽快搭出一个能稳定承接 `G1 ~ G6` 的实现骨架
2. 不为了首版速度，把未来本地优先能力堵死
3. 不为了“看起来现代”，引入过多超前复杂度
4. 让消息、请求、存储、凭据边界都可以被清楚解释

## 4. 正式选型结论

### 4.1 桌面壳层：`Tauri 2 + Web 前端`

正式结论：

- `v0.1` 采用 `Tauri 2` 作为桌面壳层
- 界面层使用 Web 技术栈承接

原因：

- 它符合桌面优先但界面仍用 Web 技术的路线
- 它比 Electron 更轻，更符合当前阶段对本地优先与复杂度克制的要求
- 它更容易对接本地数据库与系统级凭据能力

### 4.2 前端主栈：`React + TypeScript + Vite`

正式结论：

- 界面层采用 `React`
- 类型系统采用 `TypeScript`
- 前端构建与开发工具采用 `Vite`

原因：

- 当前工作台包含多栏结构、设置页、分析页和较明确的状态边界，适合组件化组织
- `TypeScript` 有助于把 `Conversation / Message / RequestRecord / UsageRecord` 等核心对象边界约束清楚
- `Vite` 足够轻，适合首版快速搭建与调试

### 4.3 路由方式：`轻量页面路由`

正式结论：

- 保留显式页面路由
- 一级页面承接 `对话工作台 / 会话分析 / 全局设置`
- 不引入过重的前端应用框架式路由体系

原因：

- 当前信息架构已经固定为 3 个一级页面
- 显式路由更利于页面边界稳定
- `v0.1` 不需要为了少量页面引入过多嵌套路由复杂度

### 4.4 状态组织：`Zustand + 领域 Store + Service + Repository`

正式结论：

- 状态层采用轻量 Store 库 `Zustand`
- 整体仍按 `领域 Store + Service + Repository` 分层

分层约束如下：

- UI 临时态留在组件内
- 会话、消息、设置、请求观测等跨组件状态进入领域 Store
- 发请求、持久化、配置拼装等流程逻辑进入 Service
- SQLite、普通配置读写、凭据访问等 IO 进入 Repository 或 Adapter

明确不做：

- 不把所有状态直接堆进一个全局大 Store
- 不让 UI 组件直接拼 Provider 请求
- 不让存储细节泄漏到页面层

### 4.5 本地持久化：`SQLite` 作为正式事实库

正式结论：

- `SQLite` 作为 `M1` 的正式事实存储
- 会话、消息、请求、usage 与普通配置，都通过统一存储层读写

原因：

- 当前模型天然是结构化数据
- `M1` 已明确不走“先纯内存后整体切换”的路线
- 未来会有筛选、排序、聚合、分析与删除语义，`SQLite` 比 JSON 文件更稳

补充约束：

- 数据访问统一走 Repository 层
- 页面层不直接接 SQL

### 4.6 敏感凭据：`Tauri` 侧系统级安全隔离

正式结论：

- `API Key` 不进入 `SQLite`
- `API Key` 不进入普通配置文件
- `API Key` 通过 `Tauri` 侧封装的系统级安全存储能力管理
- 前端只通过受控桥接能力读写凭据，不直接接触底层存储实现

原因：

- 这与现有“普通配置”和“敏感凭据”分层存储边界一致
- 它能把用户偏好设置与用户凭据清楚拆成两条数据链

实现约束：

- 普通配置里只保存非敏感状态，例如是否已配置凭据、当前使用的提供方标签等
- 真正密钥值只在系统级安全凭据层中读写

### 4.7 价格表策略：`内置静态配置 + 隐藏的高级覆盖入口`

正式结论：

- `M1` 默认采用内置静态价格表
- 同时保留隐藏的高级覆盖入口
- 高级覆盖入口不进入默认主流程，不在首版主界面高调暴露

原因：

- `M1` 需要先保证成本估算链路可用
- 价格表完全开放配置会把首版设置复杂度拉高
- 但完全写死又会堵住未来实验和校正空间

### 4.8 Provider 请求层：`OpenAI-compatible Adapter`

正式结论：

- 内部先定义统一请求对象
- 再由 Adapter 把它翻译成 OpenAI-compatible HTTP 请求
- 响应先规范化，再交给消息与观测模块

原因：

- 页面层不需要知道不同提供方的字段差异
- 后续扩展更多兼容接口时，不需要推翻上层主链
- usage 缺失、字段别名差异、错误结构差异，更容易集中兜底

### 4.9 应用重启后的遗留态清洗：`Zustand Rehydrate Hook`

正式结论：

- 应用重启后，不尝试继续恢复上一次未完成生成
- 在 `Zustand` 状态加载的 `rehydrate` 钩子中执行清洗
- 遍历 `Zustand persist` 轻量快照中的消息运行状态，只要发现 `streaming` 或 `sent`，就视为断电或异常退出遗留的僵尸状态
- `streaming` 必须立刻收尾为 `interrupted`
- `sent` 必须立刻收尾为 `error` 或等价失败态
- 清洗完成后，界面中不允许继续残留“仍在生成中”的假状态

补充说明：

- 这里清洗的是 Store 侧轻量快照中的运行时消息状态，不等于把半截助手输出落成正式历史
- 这条规则的核心目标不是“续跑”，而是“可信收尾”

### 4.10 测试策略：`单元优先，主链冒烟兜底`

正式结论：

- 核心领域对象与 Service 逻辑做单元测试
- Provider Adapter 做请求与响应规范化测试
- Repository 做基础读写测试
- 主链闭环由清晰的人工验收脚本兜底

原因：

- 当前最重要的是把主链事实边界站稳
- 首轮实现阶段不宜先铺很重的桌面端到端体系

## 5. 建议工程结构

下面是一份贴近 `M1` 的建议目录：

```text
src/
  app/
    shell/
    router/
    providers/
  domains/
    conversation/
      components/
      store/
      services/
      repository/
      types/
    settings/
      components/
      store/
      services/
      repository/
      types/
    provider/
      adapters/
      services/
      types/
    observability/
      store/
      services/
      repository/
      types/
  shared/
    ui/
    utils/
    types/
src-tauri/
  capabilities/
  storage/
  credentials/
```

这份结构想表达的重点不是目录一定长这样，而是：

- 页面壳层和领域逻辑分开
- 领域逻辑和 IO 层分开
- Web 前端和桌面原生能力边界分开

## 6. 与 `M1` 任务的映射关系

### 6.1 `G1` 项目骨架与基础运行时

优先落位：

- `app/shell`
- `app/router`
- 基础页面骨架
- 领域目录边界

### 6.2 `G2` 凭据与全局配置主链

优先落位：

- `domains/settings`
- `src-tauri/credentials`
- 普通配置 Repository

### 6.3 `G3` 会话与消息主链

优先落位：

- `domains/conversation`
- 会话列表与消息流 Store
- 会话与消息 Repository

### 6.4 `G4` 请求生命周期主链

优先落位：

- `domains/provider`
- `domains/observability`
- 负责“先建 `RequestRecord`，再发请求”的 Service

### 6.5 `G5` 本地持久化与恢复主链

优先落位：

- `SQLite` Schema
- Repository 接口与实现
- 应用启动时的恢复与清洗流程

### 6.6 `G6` 基础可观测性落地

优先落位：

- 最近一次请求观测 Store
- `UsageRecord` Repository
- 右栏观测视图

## 7. 当前定案结论

`v0.1` 进入实现阶段时，当前正式工程路线如下：

- 用 `Tauri 2` 承接桌面优先能力
- 用 `React + TypeScript + Vite` 承接工作台界面
- 用 `Zustand + 领域 Store + Service + Repository` 维持状态与边界清晰
- 用 `SQLite` 承接长期事实存储
- 用 `Tauri` 侧系统级安全隔离承接 `API Key`
- 用内置静态价格表保证 `M1` 成本估算可用，并保留隐藏高级覆盖入口
- 用 `rehydrate` 清洗规则收尾应用重启后的僵尸运行态
- 先打通非流式主链，再进入 `M2`
