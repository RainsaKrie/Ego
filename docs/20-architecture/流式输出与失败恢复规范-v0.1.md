# Ego `v0.1` 流式输出与失败恢复规范

## 1. 文档定位

本文件定义 `v0.1` 中流式输出的界面行为、数据落地边界，以及失败后的恢复策略。

它重点回答：

- 流式过程中哪些是临时态
- 哪些内容才算正式事实
- 中断、失败、手动停止后系统如何表现

大白话解释：
请求生命周期文档解决的是“大流程”，这份文档解决的是“模型正在一边吐字、一边可能出问题时，系统该怎么表现才不乱”。

## 2. 核心目标

这份规范需要同时解决四件事：

1. 定义流式过程中的 UI 状态
2. 明确临时输出和正式消息的边界
3. 规范失败后如何恢复，而不污染消息历史
4. 为首版实现控制复杂度

## 3. 核心原则

### 3.1 流式显示不等于正式落地

流式输出的文本可以边到边显示，但它在完成前都不应被视为正式稳定消息。

大白话解释：
用户可以先看到模型正在说什么，但系统不能因为“看见了几句”就默认这条回复已经完成。

### 3.2 完成态与失败态必须边界清楚

一条助手回复只有在响应完整完成后，才应进入正式消息历史。

如果中途失败：

- 可以保留失败痕迹
- 可以允许重试
- 但不能把半截输出伪装成完成消息

### 3.3 失败恢复要轻，不做复杂故障系统

`v0.1` 首版的失败恢复应聚焦于：

- 让用户知道失败了
- 保留用户消息和失败记录
- 提供轻量重试

不应一上来做：

- 多分支恢复面板
- 自动补全残缺输出
- 复杂失败队列

### 3.4 临时态只服务当前交互，不直接进入长期事实

临时态的意义是让对话体验更顺，而不是让数据层充满半成品。

## 4. 流式输出的状态分层

一次流式响应拆成 4 种状态：

### 4.1 等待中

请求已发出，模型尚未开始返回文本。

UI 应表现为：

- 明确显示正在等待响应
- 允许用户感知当前仍在等待

### 4.2 流式生成中

模型开始逐步返回文本。

UI 应表现为：

- 临时文本逐步出现
- 用户可看到回答正在形成
- 仅通过消息尾部的流动光点或细小加载态表达“尚未完成”

但这时的文本仍属于：

- 临时态
- 未完成态

正式结论：

- 不额外显示文字型“生成中”状态标识
- 临时文本与正式消息主体样式基本一致
- 未完成语义只通过尾部轻量动态信号表达

### 4.3 完成

模型完整结束响应。

UI 与数据层应表现为：

- 临时文本转化为正式助手消息
- `RequestRecord` 收尾为 `completed`
- usage 与观测信息补齐

### 4.4 失败

请求中途出错、中断、超时或未能完成。

UI 与数据层应表现为：

- 显示失败状态
- 保留失败痕迹
- 不把半截输出当正式消息
- 提供轻量重试入口

## 5. 临时输出与正式消息的边界

这是整份规范里最关键的一条。

### 5.1 临时输出

临时输出指的是：

- 模型在流式过程中逐步吐出来的文本
- 尚未确认完整结束的内容

它的定位是：

- UI 过程态
- 不等于最终事实态

### 5.2 正式消息

正式消息指的是：

- 已完整结束
- 已被系统确认稳定
- 可以进入消息历史
- 可以被后续记忆策略引用

### 5.3 当前正式结论

- 流式文本在完成前只存在于临时渲染层
- 只有在完成后才生成正式助手 `Message`
- 中途失败时不落正式助手消息

## 6. 用户在流式过程中的可执行动作

### 6.1 同一轮生成中不继续叠加新请求

首版建议：

- 不在同一会话、同一轮生成未结束时继续发起下一条新请求

原因：

- 会让消息顺序和 request 归属变复杂
- 会显著增加状态管理复杂度

### 6.2 手动停止生成

`v0.1` 首版支持手动停止生成。

正式结论：

- 生成中时，发送按钮切换为中止按钮
- 用户点击后，可主动中止当前这次流式生成
- 手动中止应被记录为独立取消态
- 不算 `completed`
- 也不等同于普通错误 `failed`

大白话解释：
这不是系统坏了，而是用户主动说“这轮到这里就停”。

## 7. 失败恢复策略

### 7.1 首版最小恢复能力

首版至少支持：

- 保留用户消息
- 保留失败 `RequestRecord`
- 如果可得，保留部分 `UsageRecord`
- 提供轻量重试按钮

补充约束：

- 用户手动中止后，半截输出同样不落正式助手消息
- 手动中止与系统失败共用“临时输出不转正”的边界规则

### 7.2 重试的真实语义

重试不是“把半截内容接着补完”。

重试的真实语义是：

- 基于原请求条件
- 发起一次新的完整请求

因此：

- 应生成新的 `RequestRecord`
- 不覆盖旧失败记录
- 不把半截内容转正

### 7.3 首版不建议做的恢复能力

以下内容不建议在 `v0.1` 首版做太重：

- 自动恢复断流
- 半截回复转草稿
- 失败内容拼接续写
- 多次失败重试队列
- 故障诊断中心

## 8. 失败后的界面表达

失败后界面至少应做到三件事：

1. 明确告诉用户这轮失败了
2. 不让用户误以为半截内容已正式保存
3. 给出轻量重试动作

推荐表达方式：

- 失败状态主提示靠近消息区
- 输入区附近或失败区域附近提供轻量重试按钮
- 悬浮提示中说明它是“重试”而非“补完”

正式结论：

- 失败状态主提示更靠近消息区
- 重试入口可在输入区附近保留轻量承接
- 不建议只把失败提示放在输入区
- 不建议做大块全局错误条

## 9. 流式重试按钮原则

失败后的重试入口应保持轻量化。

正式结论：

- 重试按钮更适合使用图标或极短文案
- 不在主界面堆长解释
- 详细语义放在悬浮提示中，例如“重试”或“按原条件重新生成”

大白话解释：
规则可以很严谨，但界面不必把严谨全部写成一句很长的按钮文案。

## 10. 当前定案结论

`v0.1` 当前正式结论如下：

- 流式输出可以临时显示，但完成前不算正式消息
- 临时文本与正式消息主体样式基本一致
- 未完成语义只通过尾部光点或细小加载态表达
- 首版支持手动停止生成
- 生成中时，发送按钮切换为中止按钮
- 手动中止属于独立取消态
- 中断、失败、手动停止都不把半截助手输出转成正式消息
- 失败后保留痕迹并允许轻量重试
- 重试基于原请求条件重新生成，并创建新的 `RequestRecord`
- 失败状态主提示靠近消息区，输入区保留轻量补救承接
