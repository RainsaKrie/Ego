# Ego `v0.1` 流式输出接入草案

## 1. 文档定位

本文件不是正式定案文档，而是进入 `M2` 前的技术评审草案。

它的作用不是直接批准开发，而是先把“流式输出要不要接、为什么现在接、接进来会改动哪里、代价是什么”讲清楚。

这份草案遵循《[架构师协议](../00-governance/架构师协议.md)》中的要求：

- 先理解，再决策
- 先形成 Spec，再进入实现
- 不用黑盒结论替代底层机制解释

## 2. 当前要解决的核心问题

当前 `M1` 已经具备：

- 多会话
- 全局默认与会话覆盖
- `API Key` 系统级安全存储
- 非流式真实请求
- `RequestRecord / UsageRecord`
- 最近一次请求观测
- 应用重启后的 `pending` 请求收尾

但当前仍缺少一个关键体验能力：

`模型在生成较长回答时，用户无法边看边等，只能等整轮请求结束后一次性看到结果。`

因此，当前草案要解决的不是“完整的生成控制系统”，而是：

`如何在不破坏现有主链可信边界的前提下，把流式文本临时显示接入当前架构。`

## 3. 为什么现在要讨论流式输出

根据已生效决策：

- `D-017`：流式输出属于 `M2`
- `D-015`：流式文本完成前只属于临时态，失败/中断时不转正式消息

所以现在讨论流式输出是合理的，因为：

1. `M1` 主链已经基本跑通，已经有稳定的数据与请求边界可承接流式
2. 流式输出是 `M2` 中最早会影响请求执行形态和 UI 状态机的能力
3. 如果不先补这份草案，后续很容易直接在现有发送链上“硬插流式”，违反《架构师协议》

## 4. 如果现在不做，会卡在哪里

如果继续停留在非流式：

- 较长回答的等待体验会比较僵
- “停止生成”能力也没有真实承载点
- 失败恢复只能围绕整轮失败做，无法处理“已经看到一部分文本但还没完成”的过程态
- 当前右栏与消息区虽有观测能力，但还缺少“生成中的运行感”

大白话解释：
不做流式不会让系统不能用，但会让它更像“请求面板”，还不够像一个对话实验台。

## 5. 它解决的核心问题是什么

流式输出真正解决的不是“更炫”，而是三个具体问题：

1. 用户可以更早看到模型开始回答，等待成本更低
2. 生成中的过程态和完成后的正式态可以被明确区分
3. 为后续“停止生成”和“按原条件重试”提供真实运行时承载点

## 6. 不做流式时的更简单替代路线

在正式接流式前，有两条更简单的替代路线：

### 6.1 保持非流式，只强化等待态

做法：

- 非流式请求不变
- 在消息区显示更明确的“等待中”占位
- 右栏补更多 pending 信息

优点：

- 最简单
- 不改请求接收形态
- 几乎不碰现有主链

缺点：

- 用户仍看不到文本逐步出现
- 无法为“停止生成”铺路
- 很难模拟真实大模型对话的生成感

### 6.2 假流式

做法：

- 后端仍用非流式
- 前端把完整文本按打字机效果分段吐出

优点：

- 接入容易
- UI 可以先看起来更顺

缺点：

- 本质是伪装
- 不支持真实中断
- 失败恢复语义失真
- 明显违反“不要黑盒推进”的项目原则

正式判断：

- `6.1` 是可接受的保守路线，但价值有限
- `6.2` 不建议采用，因为它会制造错误直觉

## 7. 当前阶段为什么真实流式属于“必要复杂度”

流式输出会增加复杂度，这是事实。

但在当前阶段，它仍然更接近“必要复杂度”，原因是：

1. 它直接属于 `M2` 已批准范围，而不是额外扩 Scope
2. 它与“停止生成”“失败恢复”“生成中观测”天然耦合，越晚接越容易返工
3. 现有 `RequestRecord / UsageRecord / 临时态不转正 / rehydrate 清洗` 已经给它准备好了关键边界

也就是说：

现在接流式不是为了做花样体验，而是为了让当前架构在 `M2` 这一阶段继续自洽。

## 8. 它会引入哪些新复杂度

这是这份草案最重要的部分之一。

### 8.1 请求接收形态从“一次性结果”变成“增量事件”

当前非流式链路是：

- 发请求
- 等完整响应
- 一次性写助手消息

流式后会变成：

- 发请求
- 接收一串增量片段
- 在前端临时渲染
- 完成后再转正式消息

### 8.2 前端要引入临时态

当前消息列表基本只展示正式消息。

流式后前端至少要多一层：

- 当前是否存在“临时助手输出”
- 当前临时文本内容是什么
- 当前生成是否已失败 / 中止 / 完成

### 8.3 后端要支持持续推送

当前 Tauri 后端是典型的 request-response 命令。

流式后要决定：

- 是让后端把增量片段通过事件推送给前端
- 还是让前端直接做流式 HTTP 接收

这会影响运行时边界和错误处理方式。

### 8.4 取消与失败的语义会更细

至少要区分：

- 还没出字就失败
- 已经出了部分字后失败
- 用户主动中止
- 正常完成

### 8.5 重启恢复会出现新的未完成态

当前重启恢复只需要清理 `pending RequestRecord` 和前端运行时轻量态。

流式后还要考虑：

- 前端可能留下半截临时文本
- 后端可能留下一个本该完成但还没收尾的流式请求

## 9. 对未来演进的影响

流式输出一旦接入，会直接影响未来这些能力：

- 停止生成
- 重试
- 失败提示靠近消息区
- usage 收尾时机
- 对话区滚动策略
- 未来的长输出与摘要策略

正面影响：

- 给后续交互能力打基础
- 让运行时更接近真实模型对话

负面风险：

- 如果临时态和正式态边界没处理好，会污染消息历史
- 如果推送方式选错，后续取消/恢复会很难收口

## 10. 当前架构中的接入点

结合当前已实现代码，流式最可能涉及以下层：

### 10.1 Rust 后端

涉及：

- `src-tauri/src/db.rs`
- `src-tauri/src/lib.rs`

需要调整：

- `send_message` 从“整轮 await”扩展为支持流式接收
- 请求执行过程中能持续向前端发增量事件
- 完成后仍由后端统一写正式 `Message / RequestRecord / UsageRecord`

### 10.2 前端工作台

涉及：

- `src/domains/conversation/components/ConversationWorkbenchPage.tsx`
- `src/domains/conversation/store/useConversationStore.ts`
- `src/domains/workspace/store/useWorkspaceRuntimeStore.ts`

需要调整：

- 消息区支持临时助手输出
- 发送按钮支持生成中态
- 完成后把临时态清掉，正式消息从持久化链接回

### 10.3 可观测性层

涉及：

- `src/domains/observability/store/useObservabilityStore.ts`

需要调整：

- pending 期间显示更合理的状态
- usage 只有在结束后才补齐
- 失败/中止时允许只部分补齐

## 11. 两种可选技术路线

### 路线 A：前端直接接收流式 HTTP

做法：

- 前端直接发起流式 HTTP 请求
- 前端逐步读取 chunk
- 后端只负责持久化或不参与这轮请求

优点：

- 路径短
- 前端能直接控制渲染节奏

缺点：

- 会绕开当前已经建立的 Tauri 后端请求主链
- `API Key` 与桌面安全边界更难统一
- 持久化与 `RequestRecord` 收尾容易分裂成两套逻辑

结论：

- 不建议采用

### 路线 B：后端接流，前端收事件

做法：

- Rust 后端保持对 Provider 的真实连接
- 后端逐块读取流式响应
- 每拿到一段增量文本，就通过 Tauri event 推给前端
- 前端只负责临时渲染
- 最终完成后仍由后端统一收尾并落正式历史

优点：

- 保持当前请求主链的一致性
- `API Key` 继续留在系统凭据边界内
- `RequestRecord / UsageRecord / Message` 仍由后端统一管理

缺点：

- 运行时实现更复杂
- 需要事件通道和前端监听管理

结论：

- 当前更推荐路线 B

## 12. 推荐路线的最小实现范围

如果后续进入批准开发，建议首轮只做以下最小范围：

1. 仅支持单轮单请求流式输出
2. 后端维持统一请求主链
3. 前端仅显示一个临时助手输出气泡
4. 完成后再落正式助手消息
5. 失败时保留失败记录，但不转正式助手消息
6. 暂不做“停止生成”
7. 暂不做多并发流式请求
8. 暂不做断点续流

这能把范围收得足够小，避免一上来就把 `M2` 撑爆。

## 13. 明确不在这轮一起做的内容

即使未来批准开发，本轮流式接入也不应顺手包含：

- 完整停止生成系统
- 自动恢复断流
- 半截回复转草稿
- 多会话并发流式调度
- 高级流式 usage 逐块统计
- 会话分析页的流式趋势展示

这些不是不做，而是不应和“先把真实流式接进主链”绑在一次决策里。

## 14. 与现有正式规范的兼容要求

如果后续批准开发，必须同时满足这些已生效约束：

### 14.1 与《请求生命周期与消息流规范》兼容

- `RequestRecord` 先建，再发请求
- 用户消息立即写入
- 完成后再落正式助手消息
- 失败时保留请求记录和用户消息

### 14.2 与《流式输出与失败恢复规范》兼容

- 临时输出不等于正式消息
- 中断、失败、手动停止都不把半截内容转正式
- 重试必须基于原请求条件重新生成

### 14.3 与《本地持久化与配置存储规范》兼容

- 临时流式态不进入 `SQLite` 正式历史
- 轻量运行态允许暂存在 `Zustand persist`
- 重启后必须能统一清洗遗留未完成态

## 15. 当前推荐结论

如果只从当前文档与代码基线出发，我的推荐是：

1. 真实流式值得做，不建议用假流式替代
2. 推荐采用“后端接流，前端收事件”的路线
3. 必须把范围收窄到“临时显示 + 完成转正式 + 失败不转正”
4. “停止生成”应单独评审，不和这轮流式接入绑死

## 16. 当前仍待确认的问题

在正式批准开发前，至少还需要再确认这些点：

1. Tauri 事件推送是否作为当前唯一流式通道
2. 临时助手输出放在现有消息流中的具体表现方式
3. 生成中时发送按钮是否先只禁用，还是直接切成“中止”
4. 流式失败时，错误提示是靠近消息区还是输入区优先
5. `UsageRecord` 在流式结束前是否完全不写，还是允许先写部分观测

## 17. 当前草案结论

按《架构师协议》的要求，当前最合理的下一步不是直接开发，而是：

- 先用这份草案完成对流式输出接入的理解与评审
- 等产品负责人确认已经理解其收益、代价、边界与替代路线
- 再决定是否把它转为正式架构文档或决策记录

也就是说：

这份草案的目标不是“催你批准”，而是确保一旦批准，我们知道自己到底在批准什么。
