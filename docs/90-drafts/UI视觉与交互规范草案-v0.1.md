# Ego `v0.1` UI 视觉与交互规范草案

## 1. 文档定位

本文件不是正式定案文档，而是 `v0.1` 在进入界面重构前的 UI 评审草案。

它解决的不是“页面有哪些区块”，而是：

- `Ego` 整体应该呈现什么气质
- 三栏实验台在视觉上如何分出主次
- 消息区、参数区、观测区分别该长什么样
- 哪些交互反馈必须统一，哪些可以延后

本草案遵循《[架构师协议](../00-governance/架构师协议.md)》：

- 先解释界面目标，再讨论实现
- 先形成规范，再进入重写
- 不靠“先做一个看着差不多的 UI”替代理解过程

## 2. 当前问题

当前实现已经有可运行界面，但它更像一套“通用桌面应用皮肤”，还不是一套被明确设计过的 `AI 对话实验台` 界面。

当前偏差主要在：

1. 视觉气质偏温和展示型，不够像专业实验台
2. 三栏虽已存在，但视觉主次还不够稳定
3. 参数区和观测区更像表单堆叠，还不像“实验控制面板”
4. 消息区虽然能看，但没有形成明确的阅读节奏和状态层级
5. 空状态、错误态、加载态、覆盖态还没有统一表达规则

大白话解释：
现在它“能用”，但不像我们想象中的 Ego。

## 3. 这份规范要解决的核心问题

这份规范真正要回答的是：

`Ego v0.1` 到底应该是一个什么气质的桌面工作台，而不是仅仅把功能摆上去。`

它至少要同时满足：

1. 看起来是一个实验台，不是普通聊天壳
2. 读消息时不累
3. 调参数时不乱
4. 看观测时不吵
5. 在桌面端长时间打开时依然稳定、克制、耐看

## 4. 如果没有这份规范，会卡在哪里

如果继续在没有 UI 规范的情况下改页面，会出现几个问题：

1. 每次都只能凭感觉修样式
2. 很容易把“结构问题”和“视觉问题”混在一起
3. 不同页面会长成不同语言
4. 后续一旦接入流式输出、停止生成、错误提示，状态表现会越来越乱
5. 任何一次“优化界面”都可能变成整页返工

## 5. 产品气质判断

`Ego v0.1` 的 UI 不应走以下两条路：

- 不应做成营销展示页式的温暖玻璃拟态
- 不应做成数据后台式的冷硬报表界面

当前更适合的方向是：

`桌面优先、专业克制、以对话为中心、以实验控制和运行观测为并列主角的 AI 工作台。`

关键词建议定为：

- `专业`
- `克制`
- `清醒`
- `高信息密度但不压迫`
- `偏工具而不是偏内容平台`

## 6. 推荐视觉方向

### 6.1 总体风格

推荐采用：

- 浅色基底
- 低饱和中性色为主
- 少量单一强调色
- 清晰边框代替大量阴影
- 轻层次代替重拟态

不推荐采用：

- 大面积奶油色和装饰性背景光斑
- 强玻璃态
- 过多圆角糖感
- 过度品牌展示感

### 6.2 视觉参考结论

如果用一句话概括，`Ego` 更应该接近：

`一套为长期工作准备的桌面实验控制台`

而不是：

`一个看起来友好的聊天玩具`

## 7. 色彩原则

### 7.1 色彩角色

建议把颜色分成 5 类角色：

1. 背景色
2. 面板色
3. 文本色
4. 边界色
5. 强调色

### 7.2 建议方向

- 背景：偏冷或中性的浅灰，不抢内容
- 面板：比背景略亮，依靠边框和分区建立层次
- 文本：深灰而不是纯黑
- 边界：明确但轻，帮助高密度信息分组
- 强调色：只保留 1 个主强调色，用于活跃导航、焦点、发送动作、关键状态

### 7.3 状态色

状态色必须统一，至少包括：

- 成功
- 进行中
- 警告
- 错误
- 已覆盖默认配置

约束：

- 状态色优先表达语义，不承担品牌装饰
- 同一状态在消息区、表单区、观测区含义一致

## 8. 字体与信息层级

### 8.1 字体目标

字体不追求“设计感很强”，优先追求：

- 中文可读
- 数字清晰
- 长时间阅读稳定

### 8.2 层级规则

至少应有以下 5 层文字层级：

1. 应用标题
2. 区域标题
3. 关键数值
4. 正文内容
5. 辅助说明和标签

约束：

- 不同层级要靠字号、字重、颜色共同拉开
- 不要大量依赖全大写英文眉标题制造层级
- 观测数值应有比普通正文更强的识别度

## 9. 布局的视觉主次

现有产品结构已定为三栏，这份规范只补视觉主次。

### 9.1 左栏

左栏应看起来像：

- 稳定导航区
- 快速切换器

不应看起来像：

- 信息主舞台
- 复杂控制中心

建议表现：

- 背景弱于中栏
- 列表节奏紧凑
- 活跃会话有明确高亮
- 覆盖默认值仅做轻标记，不在左栏解释全部细节

### 9.2 中栏

中栏必须是全页的视觉中心。

建议表现：

- 留白最多
- 文本阅读最舒服
- 环境条和输入区有稳定锚点
- 消息流明显优先于修饰元素

### 9.3 右栏

右栏应看起来像：

- 实验控制面板 + 最近一次运行观测面板

而不是：

- 一长列普通表单

建议表现：

- 分块清晰
- 标题和说明克制
- 数值信息扫描效率高
- 表单控件统一尺寸和密度

## 10. 组件级规范

## 10.1 顶部导航

建议：

- 更像应用级导航，而不是网站导航
- 标题区和页面切换区分层明确
- 活跃页面状态一眼可辨

不建议：

- 胶囊按钮太像消费级产品标签
- 导航过度抢戏

## 10.2 会话列表

建议：

- 每项高度紧凑
- 标题优先
- 次信息弱化
- 选中态显著，但不刺眼

可以包含：

- 会话标题
- 最近更新时间
- 覆盖标记

不建议：

- 每项做成大卡片
- 会话列表过于松散

## 10.3 环境条

环境条是中栏顶部的“实验条件摘要”，不是装饰条。

建议：

- 单行或双行紧凑呈现
- 重点展示模型、记忆策略、关键参数、默认/覆盖状态
- 和消息区之间形成明确分隔

## 10.4 消息气泡

消息气泡应优先服务阅读。

建议：

- 用户与助手有明确区分
- 助手消息不要做得像彩色聊天软件气泡
- 长文本段落、列表、代码块未来要能自然容纳
- 元信息弱化到不打断阅读

状态表现建议：

- `streaming`：尾部轻动态标识，不加大段提示文案
- `error` / `interrupted`：靠近消息区给出明确但克制的失败提示
- 临时消息和正式消息必须能被分辨

## 10.5 输入区

输入区必须是稳定锚点。

建议：

- 视觉上明确可编辑
- 与消息区有分隔，但不割裂
- 发送按钮是主动作
- 轻量状态读数弱化展示

生成中时：

- 如果后续接流式和停止生成，发送按钮应切换成中止动作
- 该状态应先在规范中占位，避免未来临时补丁式改动

## 10.6 参数表单

参数区不是普通设置页表单。

建议：

- 高频参数直接可见
- 标签、输入框、辅助说明形成固定节奏
- 高级参数折叠区和主参数区有视觉断层
- “恢复默认环境”应是次级但清楚的动作

## 10.7 最近一次请求观测

观测区应优先支持“扫读”，而不是阅读大段说明。

建议：

- 关键指标卡片化或分组化
- 数值层级清晰
- 错误信息单独成区，不埋在一堆指标里
- 时间、来源、usage 拆分信息放在次层

## 11. 状态与反馈规范

至少需要统一以下状态：

1. 空状态
2. 加载态
3. 发送中
4. 保存中
5. 成功反馈
6. 错误反馈
7. 已覆盖默认
8. 已恢复默认

规则：

- 主流程反馈靠近主流程出现
- 会话环境反馈靠近环境区出现
- 请求失败反馈优先靠近消息区，而不是只塞到右栏
- 不使用大面积提示条破坏主界面节奏

## 12. 密度与桌面适配

`Ego` 是桌面优先产品，因此 UI 不应过度移动端化。

建议：

- 默认桌面宽度下保持三栏
- 控件密度中高
- 消息区保留足够阅读空间
- 右栏不使用巨大控件

窄窗口时：

- 可以折叠右栏
- 但不能让整体退化成单页手机表单感

## 13. 动效原则

`v0.1` 不需要重动效，但应有少量功能性动效：

- 页面切换或区块出现时的轻过渡
- 消息新增时的轻进入
- streaming 尾部的轻动态
- 折叠区展开收起的平滑反馈

不建议：

- 大面积漂浮发光
- 明显的拟态动画
- 与实验工作台无关的装饰性 motion

## 14. 当前推荐结论

如果只基于当前阶段和产品定位，我的推荐是：

1. `Ego v0.1` 应采用专业、克制、桌面工具化的视觉方向
2. 三栏结构不改，但必须重做视觉主次
3. 右栏要从“表单堆叠”升级为“实验控制与观测面板”
4. 消息区优先保证阅读，不走花哨聊天壳路线
5. 状态语义必须统一，特别是覆盖、发送中、失败、中断

## 15. 当前仍待确认的问题

在正式定案前，还需要继续确认这些点：

1. 主强调色希望偏冷静蓝绿，还是更中性的石墨灰 + 单点强调
2. 顶部导航是否保留当前显式标签式导航，还是改成更应用化的页签结构
3. 会话列表是否允许更高密度，以换取更多可见会话
4. 最近一次请求观测是否采用“指标卡 + 明细折叠”的形式
5. 当前是否需要同时补一份低保真线框图草案

## 16. 当前草案结论

按《架构师协议》，当前最合理的下一步不是直接重写样式，而是：

- 先用这份草案确认 `Ego` 的视觉方向和交互优先级
- 再决定是否把它转为正式 `20-architecture` 或 `10-product` 规范
- 最后才进入界面重构或组件整理

也就是说：

现在要先回答“它应该长什么样”，再去回答“怎么把它写出来”。
